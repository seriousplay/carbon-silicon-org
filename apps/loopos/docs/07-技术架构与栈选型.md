# 回路OS · 技术架构与栈选型

> **状态**：v1（v4 架构转向：从飞书原生改为自建网页 SaaS）
> **决策来源**：以终为始，彻底摆脱飞书平台认证局限和 API 限制

---

## 一、架构转向的决策依据

### 放弃飞书原生的原因

| 维度 | 飞书原生的天花板 | 自建网页的解法 |
|------|---------------|--------------|
| API 限制 | 3秒回调超时/20QPS/500行查询/无SQL JOIN | 无限制，独立后端 |
| 组织可视化 | 表格+link，看不出回路层级和状态流转 | 树状图/看板/状态机可视化 |
| AI 集成 | 机器人异步消息，无法流式/实时 | 后端直接集成 LLM，流式响应 |
| 数据所有权 | 锁死飞书，迁移成本极高 | PostgreSQL 自主，可迁移可备份 |
| 状态机 | 单选字段+筛选，看不出流转 | 看板（Trello式）+ 状态图 |
| 多平台 | 仅飞书 | 未来可对接钉钉/企微/Slack |

### 保留的优势（不丢失的部分）

- **回路制方法论的编码**仍是核心价值（状态机、防双线守护、AI 教练边界、组织宪章演进）
- **飞书"在哪干活在哪治理"的理念**以另一种形式延续——网页工作台足够轻，能成为日常打开的治理入口
- **治理的公开透明**通过网页权限模型实现（全公司可见的追踪看板）

## 二、技术栈选型

```
┌─────────────────────────────────────────────┐
│  前端：Next.js 14 (App Router) + React       │
│  样式：Tailwind CSS + shadcn/ui              │
│  状态：React Query (服务端状态) + Zustand(客户端) │
│  可视化：React Flow (回路地图) + dnd-kit (看板) │
├─────────────────────────────────────────────┤
│  后端：Next.js Route Handlers / Server Actions│
│  ORM：Prisma                                  │
│  认证：自建账号（邮箱+密码 / 魔法链接）          │
│        NextAuth.js (Credentials Provider)     │
├─────────────────────────────────────────────┤
│  数据库：PostgreSQL                           │
│  托管：Neon（无服务器 PG）或 Supabase           │
├─────────────────────────────────────────────┤
│  AI：Vercel AI SDK + OpenAI/Anthropic API     │
│  定时任务：Vercel Cron / 独立 worker          │
│  邮件：Resend                                 │
├─────────────────────────────────────────────┤
│  部署：Vercel（前端+API）+ Neon（DB）          │
│  仓库：GitHub（已建 seriousplay/loopos）       │
└─────────────────────────────────────────────┘
```

### 选型理由

**Next.js 全栈**：治理工具是重表单/重列表/重状态流转的场景，Next.js 的 App Router + Server Actions 让全栈开发效率最高。Vercel AI SDK 原生集成，部署一键完成。

**Prisma**：类型安全的 ORM，schema 即文档，迁移自动化。9 主表+2 子表的复杂关系用 Prisma schema 表达最清晰，也直接对应 `01-数据模型` 文档。

**PostgreSQL（Neon）**：关系型数据库适合治理数据的强一致性需求。Neon 无服务器 PG 免费额度够初期用，分支能力方便开发。

**shadcn/ui**：复制式组件库（不是 npm 依赖），可深度定制。治理工具需要大量表格、表单、对话框、看板，shadcn/ui 的质量足够且可掌控。

**React Flow**：回路地图（Circle 树状图+角色节点）的最佳选择，支持自定义节点和连线。

**自建账号体系**：邮箱+密码（bcrypt 哈希）+ 魔法链接（免密码登录）。NextAuth.js Credentials Provider 实现。完全脱离飞书身份。

## 四、数据模型到 Prisma 的映射

`01-数据模型与表结构.md` 的 9 主表+2 子表直接映射为 Prisma 模型。关键设计：

- **归属唯一约束**：Person 表的 `homeCircleId` 是单值外键，数据库层面保证一人一回路
- **表间关联**：Prisma 的 relation 字段天然支持，比多维表格的 link 字段更强（支持级联、约束）
- **状态枚举**：PostgreSQL 的 enum 类型承载阻塞点的 10 个状态值（05 文档）
- **审计日志**：变更审计表用 Prisma 的 middleware/extension 自动记录关键变更

**优势对比多维表格**：
- 支持 SQL JOIN（多维表格不支持）→ "看回路二本周阻塞点和负责人"一次查询完成
- 支持事务（多维表格不支持）→ 状态转移 + 通知发送 + 审计记录原子化
- 支持复杂约束（多维表格不支持）→ 归属唯一、跨回路角色必有契约等约束在数据库层强制

## 五、状态机的工程实现

`05-冲突升级状态机.md` 的状态转移用**有限状态机模式**实现：

```typescript
// src/lib/statemachine.ts 伪代码
const transitions = {
  OPEN: ['ASSIGNED', 'REJECTED'],
  ASSIGNED: ['IN_PROGRESS', 'BLOCKED'],
  BLOCKED: ['IN_PROGRESS', 'ESCALATED_L0_5', 'ESCALATED_L2', 'ESCALATED_L3'],
  ESCALATED_L2: ['IN_PROGRESS', 'ESCALATED_L3'],  // 含降级
  ESCALATED_L3: ['IN_PROGRESS', 'RESOLVED', 'ESCALATED_L4'],
  // ...
}

// 转移条件量化（05 文档第六节）
const escalationRules = {
  toL2: (b) => b.slaOverdueHours > 24,
  toL3: (b) => b.escalatedAt && hoursSince(b.escalatedAt) > 48,
  toL3Repetitive: (b) => countSimilar(b, 30*24) >= 3, // 月内≥3次
}
```

**自动化的诚实边界**（呼应 05 文档第八节）：
- 定时器触发（超 SLA、48h 无更新）：cron job 自动
- 语义判断（"不可调和"、L3→L4）：仅推荐，人类确认

// src/lib/ai/tension-translator.ts

## 七、通知子系统

review 技术 P1-2 要求的统一通知框架：

| 通知类型 | 触发 | 渠道 |
|---------|------|------|
| 阻塞点 48h 无更新 | cron 定时检查 | 站内 inbox + 邮件 |
| 连续未闭环≥2 | 状态机检测 | 站内 + 邮件 + 标记治理会议程 |
| 接口 SLA 超时 | cron 定时 | 站内 + 邮件给双方回路负责人 |
| 会议提醒 | 日程 | 站内 + 邮件 |
| 升级触发 | 状态机 | 站内 + 邮件给决策者 |

**频控**：同一阻塞点 24h 内最多 2 次提醒（review 技术 P1-2）。

## 八、权限模型

review 技术 P1-3 要求。三层权限：

| 角色 | 可读 | 可写 |
|------|------|------|
| 决策者 | 全部 | 全部 + 决策记录 |
| 回路负责人 | 本回路全部 + 全公司追踪表 | 本回路的张力/阻塞点/角色/人员归属 |
| 回路成员 | 全公司追踪表（公开透明）+ 本回路详情 | 自己提的张力 |
| 横切角色 | 服务范围内 | 自己职责内的阻塞点 |

**关键原则**：追踪表全公司可见（回路制"追踪表是公开的"原则），但敏感字段（如绩效指标）按角色控制。
