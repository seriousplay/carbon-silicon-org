# 回路OS · 数据模型与表结构

> **状态**：v1（review P0-1 修正：补全字段定义 + 角色三要素 + 表间关联）
> **对应 review 问题**：技术 P0-1（零字段定义）、治理 P0-2（角色表缺 Purpose/Domain/Accountabilities）、技术 2.2.2（表间关联未定义）

---

## 一、设计原则

1. **每个字段必须可映射到飞书多维表格字段类型**——数据载体是飞书 Base，不能设计平台无法承载的字段。
2. **回路制为主，Holacracy 为辅**——角色表必须包含 Holacracy 三要素（Purpose/Domain/Accountabilities），但归属唯一是回路制硬约束，覆盖 Holacracy 的多归属灵活性。
3. **张力 ≠ 阻塞点**——review 指出两者混用。张力是"感知到的差距"（信号，未必可行动），阻塞点是"已被确认需闭环的事项"（有负责人+DDL）。两者分表，有明确流转关系。
4. **显式建模决策记录和审计**——review 指出缺失。治理会产生的决议、归属变更、角色变更必须有独立记录。

## 二、飞书多维表格字段类型映射

| 数据语义 | 飞书字段类型 | 备注 |
|---------|------------|------|
| 文本描述 | 文本 (text) | |
| 枚举（单选） | 单选 (select, multiple=false) | |
| 枚举（多选） | 多选 (select, multiple=true) | |
| 人员 | 人员 (user) | 多选模式可承载多人 |
| 关联记录 | 关联 (link) | 双向关联可自动建反向字段 |
| 自动带出他人字段 | 公式 (formula) | **默认用 formula 而非 lookup**（formula 是 lookup 超集） |
| 日期 | 日期 (datetime) | |
| 系统时间 | 创建时间 (created_at) / 修改时间 (updated_at) | 只读 |
| 系统操作人 | 创建人 (created_by) / 修改人 (updated_by) | 只读 |
| 布尔 | 复选框 (checkbox) | |
| 自动编号 | 自动编号 (auto_number) | |
| 附件（归属确认卡扫描件） | 附件 (attachment) | |

**已知平台约束**（review 技术 2.4.1 指出，设计必须遵守）：
- 单表最大列数 300
- 公式+查找引用字段合计上限 100/表
- 单表行数：商业版数万行，企业版可扩容至 200 万
- 单次查询 500 行（需分页）
- 搜索 20 QPS
- 批量创建 1000 条/次

## 三、表清单总览（1 个 Base，9 主表 + 2 子表）

> review 修正：v2 的 7 主表 + 4 子表存在张力/阻塞点混用、缺决策记录和审计日志。v1 调整为 9 主表 + 2 子表。

| # | 表名 | Holacracy 对应 | 行的语义 | 关键变化 |
|---|------|--------------|---------|---------|
| 1 | 回路 | Circle | 一行=一个回路 | 新增 父回路（支持子回路） |
| 2 | 角色 | Role | 一行=一个角色 | **补 Purpose/Domain/Accountabilities 三要素** |
| 3 | 人员 | Person | 一行=一个回路成员 | 归属唯一约束显式化 |
| 4 | **张力** | Tension | 一行=一个感知到的差距信号 | **从原"张力·阻塞点"拆出，独立成表** |
| 5 | **阻塞点（追踪表）** | Action | 一行=一个待闭环事项 | **从原"张力·阻塞点"拆出，独立成表** |
| 6 | 回路间接口 | Circle 边界 | 一行=一对回路的交付契约 | 新增 横切角色字段 |
| 7 | 指标·里程碑 | 回路健康度 | 一行=一个阶梯指标在某 Phase 的值 | |
| 8 | **决策记录** | Governance Record | 一行=治理会的一项决议 | **★ 新增（review 指出缺失）** |
| 9 | **变更审计** | — | 一行=一次归属/角色/回路结构变更 | **★ 新增（review 指出缺失）** |
| S1 | 归属确认卡（子表） | — | 一行=一张归属确认卡 | 挂在人员表 |
| S2 | 会议纪要（子表） | — | 一行=一次会议 | 含议程、决议、参与人 |

**不再用多维表格承载的**：
- 治理日志（月度）→ 飞书文档，Base 只存索引行
- 组织宪章（季度版本）→ 飞书 Wiki，Base 只存当前版本指针

### 表 4【张力 Tension】（★从原混合表拆出）

> 语义：感知到的"当前现实与潜在可能之间的差距"。是信号，**未必可行动**。

| 字段名 | 类型 | 必填 | 约束/默认 | 说明 |

| **张力类型** | 单选 | ✓ | | 值：问题性（某事卡住）/建设性（有更好做法）/澄清性（角色边界不清）。**★修正 review 指出的"阻塞点窄化张力"问题** |
| 提出人 | 人员 | ✓ | | |
| 提出时间 | 创建时间 | — | 自动 | |
| 来源 | 单选 | ✓ | | 值：战术会/治理会/机器人/表单/匿名脉冲 |
| 涉及回路 | 关联→回路 | ○ | 多值 | |
| 涉及角色 | 关联→角色 | ○ | 多值 | |
| AI翻译结果 | 文本 | ○ | | 教练Agent 翻译的结构化描述（见 06 文档，准确率目标≥85%） |
| **状态** | 单选 | ✓ | 默认"感知中" | 值：感知中/已转阻塞点/已转治理变更/已闭环/已废弃 |
| 关联阻塞点 | 关联→阻塞点 | ○ | 多值 | 转为阻塞点后回链 |
| 关联决策 | 关联→决策记录 | ○ | | 引发治理决议时回链 |
| 冲突等级 | 单选 | ○ | | 值：L0/L0.5/L1/L2/L3/L4（见 05 状态机） |

**关键流转规则**：
- 张力 → 阻塞点：当张力被确认为"有明确负责人和 DDL 的事项"时，在阻塞点表新建一行，张力状态改为"已转阻塞点"
- 张力 → 治理变更：当张力是"角色边界不清/规则过时"时，进入治理会产出决策记录，张力状态改为"已转治理变更"
- 张力可直接闭环（无需行动）：如"我误解了某规则"，状态改"已闭环"

### 子表 S1【归属确认卡】（挂在人员表）

> review 治理 P0-2 要求：归属变更需完整流程。卡片极简，只确认归属唯一性。

一人一卡。卡片内容（以人员表的字段承载，或独立小表）：

```
持卡人 / 主归属回路 / 主回路负责人
确认事项：日常优先级 / 绩效100% / 薪酬晋升建议权 / 回路去留 决定权均归主回路负责人
跨回路支援声明：可承担契约化支援角色，派活权默认归主回路负责人
本人签字 / 主回路负责人签字 / 日期
```

**重签规则**：仅归属变更时重签（走变更审计流程）；角色变更不重签。

## 六、关键约束规则汇总

| 约束 | 实现方式 | 守护机制 |
|------|---------|---------|
| 一人一主归属 | 人员表"主归属回路"是单值关联 | 多维表格字段类型强制 |
| 绩效归属=主回路负责人 | formula 字段只读 | 平台只读 |
| 跨回路角色必有契约 | 角色表"契约关联"条件必填 | 防双线守护 workflow 标红 |
| 阻塞点四要素齐全 | DDL+负责人+验收标准+描述必填 | 会议守护者会后检测 |
| 连续未闭环≥2 升级 | 阻塞点表"连续未闭环次数"字段 | 机器人定时检查 |
| 48h 无动静标红 | formula 字段 | 机器人定时检查 + @负责人 |
| 归属变更走审计 | 变更审计表 + 决策记录回链 | 流程约束 |
