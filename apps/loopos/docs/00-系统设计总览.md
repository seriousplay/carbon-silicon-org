## 一、本系统的目标

把"回路制 v2.0"的全部治理机制——归属确认、双层会议、追踪闭环、张力裁决、组织记忆、规则演进——从文档描述**落地为可运转的系统**。

诊断结论是：这类组织的核心矛盾往往是「权力-责任-资源」三角失衡，外化为部门墙 → 协作失效 → 产品掉队。回路制给出的解药是拆部门墙、按生产链路重编组为 4+1 回路。但回路制方法论通常停留在文字层面，**没有落地到任何工具载体**。回路OS 填补的正是这个空白。

**当前定位（v3 修订）**：为某 AI 团队定制的内部治理工具，把回路制真正在飞书里跑起来。不追求通用产品化、不追求跨行业适配、不追求与 Glassfrog 全球竞争。当回路制在这一个团队被验证有效后，再考虑方法论和工具的可复用性。

## 二、设计哲学（四条不变量）

1. **回路制为主，Holacracy 为辅** —— 继承回路制的归属模型和生产链路编组，吸收 Holacracy 的角色解耦、张力驱动治理和结构化会议等机制。**这是混合模型，不是 Holacracy 实现。** 当两者冲突时，回路制优先。详见上方"与 Holacracy 的 5 个关键差异"。
2. **状态在表格，交互在对话** —— 人不直接填表，而是和机器人"@回路OS"对话；所有结构化结果沉淀回多维表格。
3. **AI 做教练，不做决策者** —— Agent 承担回路制"外部教练"的三角色（会议守护者/张力翻译者/机制建设者），绝不替创始人/决策者或回路负责人拍板。**AI 能力有明确边界和降级策略，见 `06-AI能力边界与降级.md`。**
4. **为某团队深度定制，远期可复用** —— 当前阶段是为某 AI 团队定制的内部治理工具，不追求跨行业通用；"可复用产品"是回路制方法论被验证后的远期愿景，非当前定位。

## 三、系统架构：四层结构

```
┌─────────────────────────────────────────────────────────────┐
│  L4 进化层   组织宪章(季度版本化) ← 治理日志(月度/AI草稿)        │
├─────────────────────────────────────────────────────────────┤
│  L3 节奏层   战术会(30min) │ 治理会(90min) │ 战略回路(双周)     │
├─────────────────────────────────────────────────────────────┤
│  L2 感知层   张力/阻塞点 │ 冲突升级L0-L4 │ 教练Agent │ 防双线守护 │
├─────────────────────────────────────────────────────────────┤
│  L1 状态层   9主表+2子表：回路/角色/人员/张力/阻塞点/接口/指标/  │
│              决策记录/变更审计 + 归属确认卡/会议纪要              │
└─────────────────────────────────────────────────────────────┘
   自建网页 SaaS = 全部载体（状态/交互/记忆/节奏）
   技术栈：Next.js + PostgreSQL + Prisma + 自建账号 + 站内通知+邮件
```

**载体分工（v4 修正）**：
- **自建网页 = 状态 + 交互**：Next.js 全栈应用，承载 9 主表+2 子表的全部 CRUD、回路地图可视化、状态机看板、治理会工作台
- **PostgreSQL = 数据所有权**：独立数据库，数据完全自主，可迁移可备份
- **站内通知 + 邮件 = 节奏驱动**：阻塞点超时、升级触发、会议提醒通过站内 inbox + 邮件推送
- **LLM 集成 = AI 教练**：张力翻译、会议守护（会后）、治理日志起草，直接在后端集成，支持流式响应
- **飞书 = 零依赖（未来可选通知渠道）**：第一版完全不碰飞书

**为什么放弃飞书原生（v3 → v4 的关键决策）**：
1. 摆脱飞书 API 硬限制（3 秒超时/20QPS/500行/无 JOIN）
2. 获得真正的组织可视化和状态机看板（表格形态做不了）
3. AI 集成无约束（流式、实时辅助、任意 LLM）
4. 数据所有权完全自主（化解 review 三视角对"平台锁定"的所有批评）
5. 未来可多平台（不绑飞书，可对接钉钉/企微/Slack）

## 四、关键设计决策：归属 vs 角色（数据模型的灵魂）

这是回路制与 Holacracy 融合时最容易混淆的点，必须澄清：

- **归属（Accountability）** = 绩效和日常优先级由谁决定 → **唯一**（回路制 v2.0 P0-1 的硬约束，消灭双线汇报）
- **角色（Role）** = 承担什么职责 → **一人多角色**（Holacracy 把人和角色解耦的精髓）

由此推导出：
- 归属确认卡 **1 人 1 卡**，只确认归属的唯一性
- 1 个人可承担 **N 个角色**，由角色表承载（角色变更不需要重签卡，只有归属变更才重签）
- 跨回路支援角色必须是**契约化**的（经"回路间接口表"记录），派活权默认仍归主回路负责人
- 系统内置**三道防双线汇报守护**，否则消灭双线汇报的硬约束会失效

### 防双线汇报守护机制（L2 层）

| 防线 | 触发 | 行为 |
|------|------|------|
| 结构性 | 角色表里任何"跨回路支援角色"没有 link 到回路间接口契约 | 自动标红，提醒补契约 |
| 行为性 | 会议纪要/对话出现"这个我得先问一下XX领导"、消费回路越权派活、一人被两个回路负责人同时考核 | 教练Agent 翻译为张力 |
| 节奏性 | 每周治理会议程 | 自动包含"本周新增跨回路支援角色 + 契约状态"，让归属混淆台面化 |

## 五、L1 状态层：多维表格设计（核心）

### 数据表清单（1 个 Base，7 主表 + 4 子表）

| # | 表名 | Holacracy对应 | 行的语义 |
|---|------|--------------|---------|
| 1 | 回路 | Circle | 一行=一个回路（战略零 + 数据/预训练/后训练/工程基座） |
| 2 | 角色 | Role | 一行=一个角色（一人多角色，区分主归属/跨回路支援） |
| 3 | 人员 | Person | 一行=一个回路成员 + 归属确认卡状态（主归属唯一） |
| 4 | 张力·阻塞点 | Tension/Action | 一行=一个张力或阻塞点 |
| 5 | 追踪表 | 组织短期记忆 | 一行=一个待闭环阻塞点 |
| 6 | 回路间接口 | Circle 边界 | 一行=一对回路间的交付契约（含跨回路人员支援） |
| 7 | 指标·里程碑 | 回路健康度 | 一行=一个阶梯指标在某 Phase 的目标与实际 |

**4 张子表**：归属确认卡（1人1卡）、会议纪要、治理日志（月度索引）、组织宪章（季度指针）

字段详设见 `01-数据模型与表结构.md`。

## 六、L2 感知层：机器人 + 教练 Agent

### 飞书机器人「@回路OS」指令集
- **张力捕获**：`@回路OS 提张力 <描述>`（Agent翻译为结构化张力）；识别"这不是我的问题"翻译为回路间接口张力
- **查询**：`我欠什么` / `回路二本周怎么样` / `哪些事卡了两周以上`
- **节奏驱动**：战术会前自动发议程卡；阻塞点48h无更新@负责人；连续2次未闭环自动升级并通知决策者
- **归属查询**：`我归属哪个回路 / 我有哪些角色 / 谁能给我派活`

### 冲突升级 L1-L4 自动化
- L1（回路内）→ 战术会上当场到人
- L2（回路间接口延迟）→ 次日提醒+超时升L3
- L3（不可调和）→ 自动加入治理会议程标"需决策者拍板"
- L4（战略级）→ 归档战略回路

### 教练 Agent 三角色（AI 原生核心）
- **会议守护者**：检测禁用词（可能/大概/争取）、阻塞点四要素齐全、超时
- **张力翻译者**：模糊吐槽→结构化张力；含双线汇报识别
- **机制建设者**：月初从追踪表起草治理日志、季度起草宪章变更
- **硬约束**：不碰权力只建系统，不替回路负责人/决策者拍板

详设见 `02-机器人指令与Agent.md`。

## 七、L3 节奏层：双层会议体系

- **回路战术会**：每回路每周2-3次，30min硬上限
- **全链路治理会**：每周一次90min，决策者（创始人）必出席
- **战略决策回路**：每两周一次，与治理会错周

**会议纪要自动回写**：治理会后机器人从纪要提取闭环/未闭环/新阻塞/张力裁决 → 更新追踪表 → 生成摘要卡片发全员群 → 更新接口状态。

详设见 `03-会议节奏模板.md`。

## 八、L4 进化层：组织记忆与规则演进

- **治理日志（月度，AI起草）**：回路轮流复核，沉淀 Wiki
- **组织宪章（季度，版本化）**：Wiki节点，每季度一版本，≤1页，每条规则 link 到产生它的治理日志条目
- **进化闭环**：运转→张力→追踪→模式（治理日志）→规则（宪章）→回到运转，机器人每周/月/季自动驱动

## 九、可配置 + 示例预置

- **示例预置包**：5回路+人员、阶梯指标三阶表、里程碑（1月闭环率50%/3月核心指标进前5/6月综合前7/12月前5）、Plan B三路径、归属确认卡模板
- **可配置层**：回路数/名称/目的、指标体系、会议节奏、角色/张力/升级规则语义均可本地化

## 十、交付路线（三批，每批可独立运转）

| 批次 | 范围 | 可运转能力 |
|------|------|----------|
| **批次1：骨架** | L1 状态层 + L2 基础 | 组织能看见回路结构、提张力、查归属 |
| **批次2：节奏** | L3 + L2 升级 | 回路真正转起来，闭环率可追踪 |
| **批次3：进化** | L4 + Agent 完整 | 组织持续进化，教练可弹性退出 |

## 十一、与 Glassfrog 的对比（"更轻"体现在哪）

| 维度 | Glassfrog | 回路OS |
|------|-----------|--------|
| 部署 | 独立 SaaS，单独登录 | 飞书原生，零额外登录 |
| 数据载体 | 自有数据库 | 飞书多维表格（组织已有） |
| 交互 | Web 界面填表 | 飞书机器人对话 + 表格 |
| AI | 无 | 教练 Agent 三角色 |
| 进化机制 | 治理会议记录 | 治理日志 + 组织宪章 Wiki 版本化 + AI 提炼 |
| 本土化 | 英文 Holacracy 术语 | 回路制中文语义 |
| 工作流割裂 | 需在工作和治理间切换 | 在飞书里干活也在飞书里治理 |
