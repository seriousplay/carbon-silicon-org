# V6-M6-A Integrated Trial Contract and Evidence Harness

Date: 2026-07-21

Status: active slice contract

## Objective

V6-M6-A defines the minimum trustworthy trial contract for LoopOS as an
AI-native organization operating system. It does not add product behavior. It
defines what must be proven before V6-M6 can claim that a real team can use the
system from first setup to one weekly tension-to-closure operating rhythm.

## Trial Subject

The preferred trial subject is one real non-foundation-model team. If a fixture
is used before real-team operation, it must be labeled as fixture evidence and
must not be described as longitudinal real-team evidence.

Minimum real-team attributes:

- At least one organization administrator.
- At least two non-smoke human participants.
- At least one accountable human role holder for every role or AI co-assignee
  used in the trial.
- At least one operating cycle that includes a goal, tactical work, and a
  governance or structure/role clarification path.

## Required User Journey

The integrated trial must prove this path in browser-visible product surfaces:

1. Register or create an organization in `SETUP`.
2. Complete setup readiness in the locked order:
   `组织身份 / 组织结构 / 组织目标 / 角色定义 / 成员邀请 / 角色任命 / 系统配置`.
3. Activate the organization through the accepted readiness and authority gate.
4. Create the first goal cycle and a main goal for the root structure.
5. Assign at least one role to a human and, if AI is used, keep a human
   co-assignee accountable.
6. Submit or detect a candidate tension with source evidence.
7. Confirm the candidate tension by an authorized human role holder.
8. Run a tactical meeting that reviews operating health, processes a tension,
   and produces at least one owned Project or Action.
9. Run a governance meeting when structure, role, domain, accountability, or
   assignment boundaries need to change.
10. Use the Organization Brain to read organization facts and prepare or
    navigate supported actions without bypassing the relevant human authority.
11. End with evidence that generated work is owned, visible, and followable by
    the responsible human role holder.

## Evidence Classes

Evidence must stay separated. No class can stand in for another.

| Class | Proves | Does not prove |
| --- | --- | --- |
| Source/static | Implementation shape, route coverage, invariants | Runtime correctness or adoption |
| Local browser | User-visible flow and responsive behavior | Production availability |
| PostgreSQL | Authority, tenancy, lifecycle, cleanup | Browser usability |
| Production | Deployment, health, auth smoke, rollback readiness | Longitudinal team use |
| Isolation | LoopOS/BioCoach boundary and Brain reader least privilege | Product value |
| UX review | Low-friction workflow and enterprise suitability | Data correctness |
| Longitudinal real-team | Actual team use over time | Source or deployment correctness |

## Acceptance Thresholds

M6-A is accepted only when the contract and harness are independently reviewed
with no open P0/P1/P2 and the following harness inventory exists:

- One local integrated browser verifier plan or script covering the required
  journey.
- One PostgreSQL authority/isolation verifier plan or script with zero-residue
  expectations.
- One production evidence checklist covering health, readiness, authenticated
  smoke, migration status, rollback, and no secret exposure.
- One BioCoach isolation gate that requires exact denial evidence for LoopOS
  application credentials and the Brain reader.
- One longitudinal real-team verifier or runbook that names required
  participants, duration, meetings, tension outcomes, work outcomes, and
  acceptance/non-acceptance states.

## Non-Claims

Until later evidence exists, V6-M6-A does not claim:

- A real team has completed the trial.
- Production deployment has been refreshed for this milestone.
- Automatic sensing policies are active.
- AI can create formal `Tension` records without human confirmation.
- AI can execute unsupervised actions.
- BioCoach data is integrated or readable by LoopOS.
- Semantic/vector retrieval, broad notification fanout, or plugin marketplace
  expansion is complete.

## Next Slice After M6-A

If M6-A passes review, M6-B should implement the local integrated browser and
PostgreSQL evidence harness before any production deployment or longitudinal
trial claim.
