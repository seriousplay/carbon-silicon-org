## 一、执行摘要

### 1.1 V1 定位：从「设计评审的 4.41/10」到「可运行的治理骨架」

回路OS V1 是在上次三视角多智能体审查（综合评分 4.41/10）之后的一次快速工程响应。团队在约 5 小时内完成了从零到功能完备的全栈应用构建，这一速度背后反映出两个关键事实：

**正面**：V1 证明了「回路制方法论工具化」的技术可行性。Next.js + Prisma + PostgreSQL 的技术栈表现出色，64 个源文件覆盖了 9 主表 + 2 子表的完整 CRUD、10 状态的状态机、回路地图可视化、追踪看板、治理工作台。V1 已经不再是一个「概念文档」，而是一个可以在浏览器中登录、操作、流转的治理系统。

**需要正视**：V1 是一个**治理骨架而非治理肌肉**。它完成了批次 0（地基）和批次 1（状态层 CRUD），部分完成了批次 2（感知层 — 张力提交和追踪看板已可用，但 AI 翻译、升级扫描、通知系统全为空实现）。四层架构的 L1（状态层）已扎实建立，但 L2（感知层）的「感知」能力几乎为零——所有的信号检测、自动升级、AI 翻译均为 stub。

### 1.2 核心判断

| 维度 | V1 当前状态 | 距离「可运转」的差距 |
|------|-----------|-------------------|
| 组织可视化 | ✅ 回路地图、追踪看板 | 无差距，可直接使用 |
| 状态层 CRUD | ✅ 9 主表全覆盖 | 缺批量操作和导入 |
| 张力管理 | ⚠️ 可提交但无 AI 翻译 | 差一个 MAV 级别 AI 集成 |
| 冲突升级 | ⚠️ 手动转移可用，自动检测为空 | 差 cron + 通知 |
| 会议节奏 | ⚠️ 可创建会议，无议程生成/守护 | 差议程逻辑和 AI 守护 |
| AI 教练 | ❌ 三个 stub/空实现 | 差完整 MAV 方案 |
| 通知系统 | ❌ 模型已建，逻辑未写 | 差通知发送 + cron 触发 |

**结论**：V1 目前处于「单人在治理系统里操作数据」的阶段。要让一个真实组织「转起来」，至少需要补齐通知系统和 AI 翻译的 MAV。当前 V1 的定位应诚实描述为：**治理骨架验证版（Governance Skeleton MVP）**，而非「可运转的治理系统」。

### 1.3 「AI 原生组织」标签的诚实评估

在当前 V1 代码库中搜索「AI」相关实现，真实情况是：

- `translateTensionAction` — 返回固定提示文本，零 LLM 调用
- `scanEscalations` — 空数组返回，零检测逻辑
- `meetingGuardReport` — 字段已定义（Meeting 表的 `aiGuardReport`），零实现
- `package.json` — **零 AI SDK 依赖**（无 `ai`、`@vercel/ai`、`openai`、`@anthropic` 等包）
- `src/lib/ai/` — 目录不存在

**诚实回答**：「AI 原生组织」标签在当前 V1 中底气为零。这不是坏事——06 号文档本身就主张「先用程序化的确定性能力，再上需调优的 LLM 能力」。但对外沟通时必须诚实：V1 是一个**AI-ready 架构**（数据模型为 AI 预留了字段，代码为 AI 预留了 stub），而非 AI-powered 产品。

### 1.4 本报告的三个核心建议

1. **Sprint 1（2 周）：补完感知层**——接入 AI 张力翻译 MAV + 定时任务扫描升级 + 通知站内信
2. **Sprint 2（3 周）：激活节奏层**——会议议程自动生成 + 会后 AI 守护报告 + 邮件通知 + 治理日志 MAV
3. **Sprint 3（4 周）：建设进化层**——组织宪章版本化 + 治理日志模式提炼 + 北极星仪表盘 + 采纳健康度

每个 Sprint 结束后，系统都能在真实组织中被「试用」，而非等待全部完成才能上线。

## 四、AI 集成的 MAV 定义与路径

### 4.1 现状：AI 的三个「幽灵」

V1 代码库中标注了三个 AI 入口，但全都是空壳：

```
translateTensionAction  → "AI 翻译功能即将上线。当前请手动填写..."
scanEscalations         → "const signals: EscalationSignal[] = []; return signals;"
Meeting.aiGuardReport   → 数据库字段存在，零写入逻辑
```

更关键的是，`package.json` 中**没有任何 AI SDK 依赖**。这意味着 V1 不仅没有实现 AI 功能，连 AI 集成的技术基础都还没有搭建。

### 4.2 「AI 教练三角色」的诚实评估与 MAV 缩减

06 号文档定义了三角色（会议守护者、张力翻译者、机制建设者），并分了三批成熟度阶梯。但在 V1 当前状态下，追求三个角色的全覆盖是不现实的。

**MAV 原则**：最小可用版本（Minimum Actionable Version）专注于**一个角色 + 一个功能**，做到端到端可用（含降级和监控），而非三个角色的表面覆盖。

#### MAV 候选方案对比

| 方案 | 角色 | 功能 | 技术难度 | 用户感知价值 | 依赖 |
|------|------|------|---------|------------|------|
| **方案 A（推荐）** | 张力翻译者 | 模糊吐槽 → 结构化张力草稿 | ⭐⭐ | ⭐⭐⭐⭐⭐ | LLM API + prompt 工程 |
| 方案 B | 会议守护者 | 会后纪要四要素检测 | ⭐⭐⭐ | ⭐⭐⭐ | 方案 A 基础 + 信息抽取 |
| 方案 C | 机制建设者 | 月度治理日志草稿 | ⭐⭐⭐⭐ | ⭐⭐⭐ | 方案 A+B + 数据聚合 |

**推荐方案 A 的理由**：

1. **覆盖最高频场景**：提张力是回路制中最高频的治理动作。中文互联网用户习惯用模糊语言（「感觉数据回路和工程基座的协作不太顺」），AI 翻译直接解决「我不会写结构化张力」的入门障碍。
2. **技术最简单**：单次 LLM 调用（zero-shot / few-shot），不需要多步骤推理或数据聚合。成本可控（每次约 $0.001-0.005）。
3. **降级最容易**：06 文档已定义了完整的降级路径——准确率 <85% 自动切换为表单引导模式。翻译结果是「草稿」而非直接落库，容错空间大。
4. **为后续角色铺路**：翻译者的 LLM 集成模式（prompt 模板、流式响应、错误处理）可直接复用于会议守护者和机制建设者。

### 4.3 张力翻译者 MAV 技术方案

#### 技术栈选型

| 组件 | 选择 | 理由 |
|------|------|------|
| LLM 提供商 | OpenAI（GPT-4o-mini）或 Anthropic（Claude Haiku） | 中文翻译能力好，成本低 |
| SDK | Vercel AI SDK（`ai` + `@ai-sdk/openai`） | 与 Next.js 原生集成，支持流式响应 |
| 调用模式 | Server Action 中的 `generateText` | 非流式（翻译不需要实时流式展示），简化实现 |
| Prompt 策略 | Few-shot（5 个示例） | 回路制术语固定，few-shot 比微调更经济 |
| 监控 | 翻译结果存储 `aiTranslation` 字段 + 用户修改标记 | 抽检 20 条/周 |

#### 实现路径（2-3 个工作日）

**步骤 1：安装依赖和配置**

**步骤 2：实现 tension-translator.ts**

```typescript
// src/lib/ai/tension-translator.ts
import { generateText } from "ai";
import { openai } from "@ai-sdk/openai";

const TRANSLATION_PROMPT = `你是回路制治理系统的AI教练。请将以下模糊的张力描述翻译为结构化张力。

输出格式（JSON）：
{
  "structuredDescription": "结构化的张力描述（50字以内）",
  "suggestedType": "PROBLEMATIC|CONSTRUCTIVE|CLARIFYING",
  "relatedCircles": ["回路名称1", "回路名称2"],
  "confidence": 0.85
}

回路制术语说明：
- PROBLEMATIC（问题性）= 某事卡住了，需要解决
- CONSTRUCTIVE（建设性）= 有更好的做法
- CLARIFYING（澄清性）= 角色边界不清

示例：
输入："数据那边给标注需求经常延迟，我们后训练等得很焦虑"
输出：{"structuredDescription":"数据回路向预训练回路的标注交付SLA不明确，导致后训练等待时间过长","suggestedType":"PROBLEMATIC","relatedCircles":["数据回路","后训练回路"],"confidence":0.9}
`;

export async function translateTension(rawDescription: string): Promise<{
  translation: string | null;
  type: string | null;
  circles: string[];
  confidence: number;
}> {
  try {
    const { text } = await generateText({
      model: openai("gpt-4o-mini"),
      system: TRANSLATION_PROMPT,
      prompt: `请翻译以下张力描述：${rawDescription}`,
      temperature: 0.3,
    });
    const parsed = JSON.parse(text);
    return {
      translation: parsed.structuredDescription || null,
      type: parsed.suggestedType || null,
      circles: parsed.relatedCircles || [],
      confidence: parsed.confidence || 0,
    };
  } catch (error) {
    console.error("AI 翻译失败:", error);
    return { translation: null, type: null, circles: [], confidence: 0 };
  }
}
```

**步骤 3：修改 translateTensionAction**

将 stub 替换为调用 `translateTension()`，返回翻译结果 + confidence。前端以「草稿」模式展示，用户确认后落库。

**步骤 4：实现降级监控**

- 记录每次翻译的 confidence
- 计算周准确率：`(用户接受翻译未修改的次数) / 总翻译次数`
- 低于 85% 时切换为表单引导模式

#### 成本估算

| 用量 | 月成本 |
|------|--------|
| 日均 20 次翻译 | ~600 次/月 × $0.00015（GPT-4o-mini input）+ $0.0006（output）≈ $0.5-1 |
| 含重试和降级 | < $5/月 |

### 4.4 后续角色路径

MAV 成功后，按以下顺序渐进增强：

1. **v1.1**：会议守护者—会后纪要四要素检测（复用翻译者的 LLM 调用模式，但 prompt 改为信息抽取）
2. **v1.2**：治理日志草稿（聚合翻译者 + 守护者的数据，用 LLM 做月度摘要）

### 5.2 通知 / 定时任务 MAV 方案

#### 触发场景优先级

| 优先级 | 场景 | 触发 | 渠道 | 频控 |
|--------|------|------|------|------|
| P0 | 阻塞点 48h 无更新 | cron 每小时检查 | 站内 Notification | 同一阻塞点 24h 内最多 1 次 |
| P1 | 连续 2 次未闭环 | 状态转移时检测 | 站内 + 标记治理会议程 | 每阻塞点每周期最多 1 次 |
| P2 | 升级触发（L2/L3/L4） | 状态转移时检测 | 站内 | 即时，无频控 |
| P3 | 会议提醒 | 会前 30min | 站内 + 邮件 | 每会议 1 次 |

#### 实现路径

**方案：Vercel Cron Jobs + 站内 Notification**

```
┌──────────────────────────────────────────┐
│            Vercel Cron Jobs              │
│  GET /api/cron/scan-escalations (每小时)  │
│  GET /api/cron/meeting-reminders (每10分钟)│
└──────────────┬───────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────┐
│          Server Action / API Route        │
│  1. 查询符合条件的阻塞点/会议              │
│  2. 调用 statemachine.detectEscalation()  │
│  3. 写入 Notification 表                  │
│  4. （可选）发送邮件                      │
└──────────────────────────────────────────┘
```

**为什么不选独立 Worker**：V1 规模下（预计 <50 用户），Vercel Cron 免费额度（每天 1 次免费执行，Pro 套餐支持每分钟）完全够用。独立 Worker 增加运维复杂度，待用户规模 >100 或需要 <1 分钟频率时再迁移。

**步骤 1**：创建 `src/app/api/cron/scan-escalations/route.ts`

```typescript
import { prisma } from "@/lib/db";
import { detectEscalation, isOverdue } from "@/lib/statemachine";
import { NextResponse } from "next/server";

export async function GET(request: Request) {
  // Vercel Cron 鉴权
  const authHeader = request.headers.get("authorization");
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }

  const organizations = await prisma.organization.findMany({ select: { id: true } });

  for (const org of organizations) {
    // 1. 48h 无更新的阻塞点
    const staleBlockers = await prisma.blocker.findMany({
      where: {
        organizationId: org.id,
        status: { notIn: ["RESOLVED", "REJECTED"] },
      },
      include: { owner: true },
    });

    for (const blocker of staleBlockers) {
      if (isOverdue(blocker.updatedAt, blocker.status)) {
        // 检查 24h 内是否已通知过
        const recentNotification = await prisma.notification.findFirst({
          where: {
            recipientId: blocker.ownerId,
            type: "blocker_overdue",
            createdAt: { gte: new Date(Date.now() - 24 * 3600000) },
          },
        });
        if (!recentNotification) {
          await prisma.notification.create({
            data: {
              organizationId: org.id,
              recipientId: blocker.ownerId,
              type: "blocker_overdue",
              title: `阻塞点超时：${blocker.description.slice(0, 50)}`,
              body: `你的阻塞点已超过 48 小时未更新。请前往追踪看板处理或重新确认截止日期。`,
            },
          });
        }
      }

      // 2. 升级信号检测
      const signal = detectEscalation({
        status: blocker.status,
        hoursSinceUpdate:
          (Date.now() - blocker.updatedAt.getTime()) / 3600000,
        similarCountThisMonth: 0, // TODO: 同类统计
      });
      if (signal) {
        // 创建通知或自动升级...
      }
    }
  }

  return NextResponse.json({ success: true });
}
```

交付后综合完成度    ~55%                 ~75%                 ~95%

## 七、北极星指标定义

### 7.1 北极星指标的选择原则

回路制治理系统的成功不取决于「系统有多少功能」，而取决于「组织是否真的在用它在治理」。因此，北极星指标必须反映**行为改变**而非**功能使用**。

参考以下评估框架：

| 候选指标 | 优点 | 缺点 | 适合作为北极星？ |
|---------|------|------|----------------|
| DAU/WAU | 简单直接 | 登录 ≠ 治理；不能区分「刷一眼」和「真正使用」 | ❌ 太浅 |
| 闭环率 | 直接衡量治理有效性 | 可能引发只提容易闭环的张力 | ⚠️ 需要配合张力复杂度衡量 |
| 张力提出率 | 反映组织透明度和心理安全 | 数量 ≠ 质量 | ⚠️ 需要配合闭环率 |
| 会议出席率 | 反映治理节奏的健康度 | 出席 ≠ 参与 | ⚠️ 需要配合会议质量指标 |
| **治理密度** | 综合衡量 | 需要定义和校准 | ✅ 推荐 |

### 7.2 推荐北极星：治理密度分数（Governance Density Score, GDS）

**定义**：

```
GDS = (闭环率 × 0.4) + (人均周张力数 × 0.25) + (会议按时完成率 × 0.2) + (采纳健康度 × 0.15)
```

**各子指标定义**：

| 子指标 | 计算方式 | 目标（3 个月后） |
|--------|---------|----------------|
| **闭环率** | 本周闭环阻塞点数 / 本周应闭环阻塞点数 | ≥60% |
| **人均周张力数** | 本周总新增张力数 / 活跃用户数 | ≥1.5（表明组织在持续感知问题） |
| **会议按时完成率** | 按时完成（≤30min战术会/≤90min治理会）的会议数 / 总会议数 | ≥80% |
| **采纳健康度** | 复合分（登录频率 ×0.3 + 张力提交 ×0.3 + 阻塞点更新 ×0.2 + AI 翻译确认 ×0.2） | ≥70 |

**为什么用复合指标而非单一指标**：

单一指标容易被「玩弄」（Goodhart's Law）。复合指标降低作弊可能性——你很难同时刷高闭环率、张力提出率和会议出席率。

**反作弊护栏**：
- 闭环率若 >90% 且人均张力数 <0.5 → 标记「刷闭环」告警
- 人均张力数若 >5 且闭环率 <20% → 标记「张力倾泻」告警
- AI 翻译确认率若 >95% 且修改率 >60% → 标记「盲点确认」告警

### 7.3 V1 阶段的观测指标（Sprint 1 立即可用）

| 指标 | 数据源 | 用途 | 可追踪时间 |
|------|--------|------|----------|
| 活跃回路数 | Circle 表 count | 组织覆盖度 | ✅ 当前 |
| 感知中张力数 | Tension 表 status=SENSING count | 问题感知活跃度 | ✅ 当前 |
| 待闭环阻塞数 | Blocker 表 status≠RESOLVED/REJECTED count | 工作负载 | ✅ 当前 |
| 闭环率（7 天） | Blocker 表 7 天内 RESOLVED/ 总数 | **核心有效性** | Sprint 1 |
| 人均张力数 | Tension 表 7 天内新增 / 活跃用户数 | **治理参与度** | Sprint 1 |
| 会议按时完成率 | Meeting 表 durationMin ≤ 上限 / 总数 | **会议纪律** | Sprint 2 |
| AI 翻译采纳率 | aiTranslation 被用户确认未修改 / 总翻译数 | **AI 可信度** | Sprint 1 |
| 通知响应率 | Notification readAt 非空 / 总数 | **通知有效性** | Sprint 1 |

### 7.4 用户验证的实验设计

在 Sprint 1 完成后（系统首次具备「感知」能力），建议进行 2 周的受控实验：

**实验组**：一个真实回路（如 3-5 人），要求所有工作阻塞点通过回路OS 追踪
**对照组**：维持现有工作方式（飞书群/口头/Excel）

**观测指标**：
1. 阻塞点从「被发现」到「被闭环」的平均时间（TTR — Time to Resolution）
2. 阻塞点被「遗漏」的数量（对照组通过访谈收集）
3. 团队成员对「协作透明度」的 NPS 评分

**成功阈值**：实验组 TTR **≤ 对照组 TTR × 0.8**，且 NPS 不下降。

## 八、行业标杆路径评估

### 8.1 行业参照系

| 产品 | 定位 | 核心用户 | AI 集成度 | 回路OS 对标策略 |
|------|------|---------|----------|---------------|
| **Glassfrog** | Holacracy 官方工具 | Holacracy 采纳组织 | FrogBot（AI 会议助手，2025 上线） | 差异化：回路制（非 Holacracy）+ 中文原生 + 内部工具定位 |
| **Asana + 自定义** | 通用项目管理 | 任何团队 | Asana Intelligence | 差异化：回路制专属语义（张力/阻塞点/升级）vs 通用任务 |
| **飞书多维表格 DIY** | 用飞书搭治理表格 | 中国科技公司 | 无 AI 治理能力 | 差异化：专用系统 > 通用表格；状态机 + AI |
| **自建 Notion 系统** | Notion 模板治理 | 早期创业公司 | Notion AI（通用） | 差异化：关系数据库 + 回路可视化 + 专业状态机 |

### 8.2 回路OS 的差异化路线

回路OS 不做「又一个项目管理工具」，而是做「回路制方法论的专属操作系统」。差异化护城河有三层：

**第一层（短期，Sprint 1-2）：方法论编码**
- 回路制的专属语义（张力 vs 任务、阻塞点 vs bug、升级 L0-L4 vs 优先级 P0-P3）
- 防双线守护的自动化检测（通用工具做不到）
- 归属确认卡（Holacracy 和回路制的核心差异点）

**第二层（中期，Sprint 3+）：AI 教练闭环**
- 张力翻译 + 会议守护 + 治理日志 → 三个角色的数据共享（06 文档的协作协议）
- 随着数据积累，AI 的模式提炼能力从「基于规则」升级为「基于本组织历史模式」

**第三层（长期，12 个月+）：方法论验证的社会证明**
- 如果在 1 个组织中闭环率从基线提升到 60%+，回路制的工具化就得到了验证
- 此时可以将「内部工具」升级为「可配置的治理 OS」，服务更多 AI 团队
- 北极星从「单个组织的治理密度」扩展为「采纳组织的治理密度中位数」

### 8.3 「行业标杆」的现实时间线

| 时间点 | 里程碑 | 对标 |
|--------|--------|------|
| 当前（V1） | 治理骨架可用 | 比 Glassfrog 少了 AI 和会议管理，但回路语义更贴近中国团队 |
| Sprint 1 完成（+2 周） | AI 张力翻译 + 通知系统上线 | 在「AI 辅助治理」维度上首次超越自建表格方案 |
| Sprint 2 完成（+5 周） | 会议守护者 + 治理日志 MAV | 基本追平 Glassfrog 的治理能力（不含宪章） |
| Sprint 3 完成（+9 周） | 宪章版本化 + 北极星仪表盘 | 在「治理数据分析」维度上开始超越 Glassfrog |
| 6 个月 | 1 个组织验证成功（闭环率 ≥60%） | 可对外宣称「验证过的回路制治理 OS」 |
| 12 个月 | 3 个组织的案例积累 | 可启动从「内部工具」到「产品化」的转型 |

| 阶段 | 可用标签 | 条件 |

### 9.2 V1 的优势（值得保留和放大）

1. **技术地基扎实**：Next.js + Prisma + PostgreSQL 的选型在 V1 中被证明高效。5 小时完成 64 个文件的构建证明了 DX（开发者体验）优秀。
2. **数据模型忠实于方法论**：693 行 Prisma schema 完整映射了回路制的核心概念——归属唯一、角色解耦、张力/阻塞点分离、回路间接口契约、冲突升级全路径。
3. **状态机是隐藏的技术资产**：10 状态 + 26 条转移 + `detectEscalation` 函数是整个系统自动化的核心引擎。虽然当前只用于手动转移校验，但它的设计已经为完全自动化做好了准备。
4. **美学语言差异化**：「活体组织美学」（种子/生长/成熟/需求光明）在 B2B 工具中极为罕见。首页的「组织呼吸中」动态指示器、空状态的诗意引导，是难以量化的用户情感资产。
5. **06 号文档的诚实性**：在上次审查的 AI 过度承诺批评之后，06 号文档是所有修正中最诚实的一份——它「先说了不能做什么」，定义了量化目标和降级策略。这份文档是接下来所有 AI 开发的路标。

### 9.3 V1 的不足（下一阶段的重点进攻方向）

1. **感知层缺失**（最关键）：系统的「感知」能力全部依赖人类手动操作。没有自动检测、没有通知、没有 AI 辅助。用户容易「忘记来回路OS 看一眼」。
2. **AI 集成零进展**：三个 stub 是三个技术债务。不仅是功能缺失，更是**没有验证过 LLM 在回路制场景下的实际表现**。06 号文档的「准确率 ≥85%」目标至今是假设。
3. **节奏层机械而非有机**：会议模块是 CRUD 而非治理节奏引擎。没有议程生成、没有时间提醒、没有会后分析。一个「治理系统」不会主动提醒你该开会了。
4. **没有 Onboarding**：新用户（尤其是非技术用户）面对 9 张主表和回路制术语，不知所措。首页的空状态引导是好的种子，但远不足以引导一个组织完成从 0 到 1 的治理启动。
5. **测试覆盖近乎为零**：唯一有测试的是 `statemachine.ts`。所有 Server Actions、页面组件、数据查询都无测试。在快速迭代中这是合理的取舍，但在 Sprint 1 前后应开始补齐关键路径的测试。

| # | 行动 | 工作量 | 理由 |

#### Sprint 1 推进（2 周）

严格按第六节的 Sprint 1 规划执行。如果资源紧张，砍掉 D1.4（防双线守护）和 D1.6（接口 UI），优先保证 D1.1-D1.3 + D1.5。

#### 中期方向（Sprint 2-3）

### 9.5 给创始团队的一句话

> V1 用 5 小时证明了回路制可以工具化。接下来 9 周要证明的是：这个工具真的能让组织运转得更好。不要被「AI 原生组织」的标签诱惑而跳过验证步骤——先用最简 AI（MAV 翻译者）跑通一个真实回路的闭环，再谈行业标杆。**验证比功能多更重要，闭环率比代码行数更重要。回路制方法论在真实组织中的治理密度分数（GDS），是回路OS 唯一的北极星。**
