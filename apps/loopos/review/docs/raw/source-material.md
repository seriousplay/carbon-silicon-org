## 文件 1：README.md

# 回路治理组织操作系统（回路OS）

> **定位**：一个飞书原生的、比 Glassfrog 更轻、AI 增强的组织治理操作系统。
> 把"回路制 v2.0"的全部治理机制——归属确认、双层会议、追踪闭环、张力裁决、组织记忆、规则演进——从文档描述落地为**可运转的系统**。

## 为什么

很多快速扩张的 AI 团队都会撞上同一个矛盾：「权力-责任-资源」三角失衡，外化为部门墙 → 协作失效 → 产品掉队。"回路制"给出的解药是把部门墙拆掉、按真实生产链路重编组为 4+1 个回路，并配以双层会议、追踪表、组织宪章等治理机制。

但这些机制全部停留在**文档描述层**，没有落地到任何工具载体。飞书在文档里只被当作"群"和"文档"用过，多维表格的治理能力完全没被激活。

**回路OS 要填补的，正是这个空白**——让回路制从"咨询方案的文字"变成"组织每天在飞书里呼吸的系统"。

## 核心理念（四条不变量）

1. **回路制为表，Holacracy 为里** —— 表层用本土词（回路/阻塞点/治理会），底层对齐 Holacracy 的 Role/Circle/Tension 数据结构。
2. **状态在表格，交互在对话** —— 人和"@回路OS"对话，结构化结果沉淀回多维表格。
3. **AI 做教练，不做决策者** —— Agent 承担外部教练三角色（会议守护者/张力翻译者/机制建设者），绝不替回路负责人或决策者拍板。
4. **可配置 + 示例预置** —— 底层通用，开箱即用一套 4+1 回路模板和阶梯指标（首版预置源自某大模型团队的真实生产链路）。

## 系统架构（四层）

```
┌─────────────────────────────────────────────────────────────┐
│  L4 进化层   组织宪章(季度/Wiki版本化) ← 治理日志(月度/AI草稿)  │
├─────────────────────────────────────────────────────────────┤
│  L3 节奏层   战术会(30min) │ 治理会(90min) │ 战略回路(双周)     │
├─────────────────────────────────────────────────────────────┤
│  L2 感知层   张力/阻塞点 │ 冲突升级L1-L4 │ 教练Agent │ 防双线守护 │
├─────────────────────────────────────────────────────────────┤
│  L1 状态层   7张主表：回路/角色/人员/张力-阻塞/追踪/接口/指标    │
└─────────────────────────────────────────────────────────────┘
   飞书多维表格(Base)=状态  机器人@回路OS=交互  文档/Wiki=记忆  日历/会议=节奏
```

## 关键设计决策

### 归属 vs 角色（数据模型的灵魂）

- **归属（Accountability）** = 绩效和日常优先级由谁决定 → **唯一**（回路制 v2.0 P0-1 的硬约束，消灭双线汇报）
- **角色（Role）** = 承担什么职责 → **一人多角色**（Holacracy 把人和角色解耦的精髓）

所以：
- 归属确认卡 **1 人 1 卡**，只确认归属的唯一性
- 1 个人可承担 **N 个角色**，由角色表承载
- 跨回路支援角色必须是**契约化**的（经"回路间接口表"记录），派活权默认仍归主回路负责人
- 系统内置**三道防双线汇报守护**，否则消灭双线汇报的硬约束会失效

## 一、本系统的目标

把"回路制 v2.0"的全部治理机制——归属确认、双层会议、追踪闭环、张力裁决、组织记忆、规则演进——从文档描述**落地为可运转的系统**。

### 载体分工
- **多维表格 = 状态**：7 张主表 + 4 张子表，是组织的"当前快照"
- **机器人 = 交互**：提张力、查阻塞、催闭环、生成议程，背后读写表格
- **文档/Wiki = 记忆**：组织宪章（季度版本化）、治理日志（月度）、技术路线决策备忘录

## 四、关键设计决策：归属 vs 角色

- **归属（Accountability）** = 绩效和日常优先级由谁决定 → **唯一**
- **角色（Role）** = 承担什么职责 → **一人多角色**

由此推导：
- 归属确认卡 1 人 1 卡
- 1 人 N 角色，角色变不重签卡
- 跨回路支援必须契约化（回路间接口表记录）
- 三道防双线汇报守护

### 防双线汇报守护机制（L2 层）

| 防线 | 触发 | 行为 |
|------|------|------|
| 结构性 | 跨回路支援角色无契约 link | 自动标红，提醒补契约 |
| 行为性 | 对话出现"得先问XX领导"、越权派活 | 教练Agent 翻译为张力 |
| 节奏性 | 每周治理会议程 | 自动包含本周新增跨回路支援角色+契约状态 |

## 五、L1 状态层：多维表格设计

### 数据表清单（1 个 Base，7 主表 + 4 子表）

| # | 表名 | Holacracy对应 | 行的语义 |
|---|------|--------------|---------|
| 1 | 回路 | Circle | 一行=一个回路 |
| 2 | 角色 | Role | 一行=一个角色 |
| 3 | 人员 | Person | 一行=一个回路成员+归属确认卡状态 |
| 4 | 张力·阻塞点 | Tension/Action | 一行=一个张力或阻塞点 |
| 5 | 追踪表 | 组织短期记忆 | 一行=一个待闭环阻塞点 |
| 6 | 回路间接口 | Circle 边界 | 一行=一对回路间的交付契约 |
| 7 | 指标·里程碑 | 回路健康度 | 一行=一个阶梯指标在某 Phase 的目标与实际 |

**4 张子表**：归属确认卡、会议纪要、治理日志、组织宪章

## 六、L2 感知层：机器人 + 教练 Agent

### @回路OS 指令集
- 张力捕获：`@回路OS 提张力 <描述>`
- 查询：`我欠什么` / `回路二本周怎么样` / `哪些事卡了两周以上`
- 节奏驱动：战术会前自动发议程卡；48h无更新@负责人；连续2次未闭环自动升级
- 归属查询：`我归属哪个回路 / 我有哪些角色 / 谁能给我派活`

### 冲突升级 L1-L4 自动化
- L1（回路内）→ 战术会上当场到人
- L2（回路间接口延迟）→ 次日提醒+超时升L3
- L3（不可调和）→ 自动加入治理会议程标"需决策者拍板"
- L4（战略级）→ 归档战略回路

### 教练 Agent 三角色
- **会议守护者**：检测禁用词、阻塞点四要素、超时
- **张力翻译者**：模糊吐槽→结构化张力；含双线汇报识别
- **机制建设者**：月初起草治理日志、季度起草宪章变更
- **硬约束**：不碰权力只建系统，不替回路负责人/决策者拍板

## 七、L3 节奏层：双层会议体系

- 回路战术会：每回路每周2-3次，30min硬上限
- 全链路治理会：每周一次90min，决策者必出席
- 战略决策回路：每两周一次，与治理会错周

## 八、L4 进化层：组织记忆与规则演进

- 治理日志（月度，AI起草）：回路轮流复核，沉淀 Wiki
- 组织宪章（季度，版本化）：Wiki节点，每季度一版本，每条规则 link 到产生它的治理日志条目
- 进化闭环：运转→张力→追踪→模式（治理日志）→规则（宪章）→回到运转

## 九、可配置 + 示例预置

- 示例预置包：5回路+人员、阶梯指标三阶表、里程碑、Plan B三路径、归属确认卡模板
- 可配置层：回路数/名称/目的、指标体系、会议节奏、角色/张力/升级规则语义均可本地化

## 十一、与 Glassfrog 的对比

| 维度 | Glassfrog | 回路OS |
|------|-----------|--------|
| 部署 | 独立 SaaS，单独登录 | 飞书原生，零额外登录 |
| 数据载体 | 自有数据库 | 飞书多维表格（组织已有） |
| 交互 | Web 界面填表 | 飞书机器人对话 + 表格 |
| AI | 无 | 教练 Agent 三角色 |
| 进化机制 | 治理会议记录 | 治理日志 + 组织宪章 Wiki 版本化 + AI 提炼 |
| 本土化 | 英文 Holacracy 术语 | 回路制中文语义 |
| 工作流割裂 | 需在工作和治理间切换 | 在飞书里干活也在飞书里治理 |
