### 1.3 当前阶段定位

V2 已从 V1 的「治理骨架验证版（Governance Skeleton MVP）」跃迁至：

> **「AI 增强的回路制治理系统（AI-Augmented Governance System）」**
>
> 距离「可对外展示的产品」还有一层窗户纸——核心功能完整，但缺少打磨（Onboarding、测试、性能优化、文档）。

在技术成熟度阶梯上的位置：

```
概念原型 ──► 技术验证 ──► 功能完整 ──► 可对外展示 ──► 产品化 ──► 规模化
   (V0)        (V1)        (V2) ← 我们现在在这里
```

**V2 已跨越「功能完整」的门槛。**系统可以做以下事情：
- 用真实 LLM 翻译张力
- 自动扫描超时并通知
- 生成会议议程
- AI 起草治理日志
- 版本化组织宪章
- 追踪闭环率和采纳健康度

这在 V1 中全部是空白。

## 三、「AI 原生组织」标签底气评估

### 3.1 V1 的诚实评估回顾

V1 战略报告明确指出：

> 「AI 原生组织」标签在当前 V1 中底气为零。V1 是一个 AI-ready 架构，而非 AI-powered 产品。

当时的证据：`package.json` 零 AI SDK 依赖，三个 stub 空实现，`src/lib/ai/` 目录不存在。

### 3.2 V2 的真实底气

**AI 教练三角色的代码实现状态**：

| 角色 | 实现 | 质量 | LLM 集成 |
|------|------|------|---------|
| **张力翻译者** | `tension-translator.ts` 67 行 | ⭐⭐⭐⭐ | OpenAI/Anthropic/StepFun，含降级 + JSON 解析容错 |
| **会议守护者** | `meeting-assistant.ts` 96 行 | ⭐⭐⭐⭐ | 议程生成 + 会后守护报告（禁用词/四要素/超时） |
| **机制建设者** | `pattern-builder.ts` 126 行 | ⭐⭐⭐⭐ | 月度日志起草 + 模式提炼 + 可信度评分 |

**AI 基建层**：

- `provider.ts`：三 provider 抽象（OpenAI / Anthropic / StepFun），含 `askAI` + `streamAI` 两个统一接口
- `isAIAvailable()` 运行时检测
- `package.json` 真实依赖：`ai ^7.0.16`、`@ai-sdk/openai`、`@ai-sdk/anthropic`
- AI 翻译与创建张力 Server Action 已打通（新增张力页有「✨ 让 AI 帮我结构化」按钮）

### 3.3 分层底气评估

| 维度 | V1 | V2 | 结论 |
|------|----|----|------|
| AI 基础设施 | 0/10 — 零 SDK 依赖 | **8/10** — 多 provider 抽象、降级、流式 | AI-ready 已升级为 AI-powered |
| 张力翻译 | 0/10 — 固定提示文本 | **8/10** — 真实 LLM 调用，含草稿确认机制 | 核心差异化已兑现 |
| 会议 AI | 0/10 — 字段存在零实现 | **7/10** — 议程生成+守护报告，但缺实时守护 | 追上设计承诺的 70% |
| 治理日志 AI | 0/10 — 无 | **7/10** — 月度起草+模式提炼+可信度评分 | Sprint 3 核心交付 |
| AI 原生体验 | 0/10 — 无 AI 交互 | **6/10** — AI 按钮嵌入工作流，但无对话式交互 | 功能驱动而非体验驱动 |
| **综合 AI 底气** | **0/10** | **7.2/10** | 从「零底气」到「有底气但不完美」 |

### 3.4 标签使用的诚实建议

回顾 V1 战略报告中定义的标签阶梯：

| 阶段 | 标签 | V2 是否达到 |
|------|------|-----------|
| V1 | 「回路制治理工具」 | — |
| Sprint 1 完成 | 「AI 辅助的回路制治理工具」 | ✅ **已达到** |
| Sprint 2 完成 | 「AI 教练驱动的治理 OS」 | ✅ **已达到** |
| Sprint 3 完成 | 「**AI 原生组织治理 OS**」 | ⚠️ **接近但未完全达到** |

**V2 现在可以诚实使用的标签**：「AI 教练驱动的回路制治理系统」

不建议直接使用「AI 原生组织治理 OS」，因为：
1. 「AI 原生」暗示 AI 是系统不可分割的核心而非附加层——当前 AI 更像是「嵌入工作流的辅助工具」而非「系统原生能力」
2. 缺失对话式交互（`@回路OS` 机器人概念停留在文档），交互仍以表单为主
3. AI 三角色的协作协议（06 文档定义的）代码虽有但尚未在真实组织数据上验证
4. 没有 AI 使用数据（翻译准确率、采纳率、守护报告质量），无法证明「AI 原生」的成效

**诚实标签**：「AI 教练驱动的回路制治理系统」（AI-Coach-Powered Loop Governance System）

对外沟通建议：强调「三个 AI 角色嵌入治理全流程——从提张力到开会到月度复盘」，而不是空泛的「AI 原生」。

## 四、行业标杆差距矩阵

### 4.1 与 Glassfrog 的对比（2026 年最新）

Glassfrog 在 2025 年发布了 FrogBot v2 AI 会议助手。以下是回路OS V2 与 Glassfrog 的逐维对比：

| 维度 | Glassfrog（最新） | 回路OS V2 | 差距 | 评价 |
|------|------------------|-----------|------|------|
| **治理方法论** | Holacracy 标准 | 回路制（Holacracy 为辅） | — | 不同流派，无优劣 |
| **AI 张力翻译** | ❌ 无 | ✅ 三种类型分类 + 结构化草稿 | **领先** | 回路OS 独特优势 |
| **AI 会议助手** | FrogBot v2（议程+纪要） | 议程生成 + 会后守护报告 | 持平 | FrogBot 实时性更强，回路OS 纪律性更严 |
| **AI 治理日志** | ❌ 无 | ✅ 月度起草 + 模式提炼 | **领先** | 回路OS 独特优势 |
| **组织宪章** | 基本版本记录 | 版本链 + 批准/归档工作流 | 持平 | 均有基本版本管理 |
| **回路可视化** | 基本列表 | React Flow 环形地图 + 弹性物理布局 + 圈层视角 | **领先** | 回路OS 可视化是差异化优势 |
| **通知系统** | 邮件通知 | 站内 inbox + 邮件 + 频控 | **领先** | 回路OS 站内通知体验更好 |
| **搜索** | 基本搜索 | ⌘K 全局搜索 + 防抖 | 持平 | 功能相似 |
| **移动端** | 响应式 Web | Sheet 导航 + 响应式布局 | 持平 | 功能相似 |
| **Onboarding** | 引导流程 | 一键初始化模板 + 空状态引导 | 略逊 | Glassfrog 引导更成熟 |
| **中文支持** | ❌ 英文为主 | ✅ 中文原生 | **大幅领先** | 回路OS 核心壁垒 |
| **数据主权** | Glassfrog 云 | PostgreSQL 自托管 | **领先** | 回路OS 数据完全自主 |

**关键洞察**：回路OS 在 AI 覆盖度（三个角色 vs Glassfrog 的一个半）和中文原生体验上已经超越 Glassfrog。但在产品打磨（Onboarding、文档、稳定性）上仍有差距。

### 4.2 与 Linear 的对比（交互标杆）

Linear 不是治理工具，但代表「开发者工具应有的交互标准」：

| 维度 | Linear | 回路OS V2 | 差距 |
|------|--------|-----------|------|
| 键盘快捷键 | 全面（⌘K 无处不在） | 仅 ⌘K 搜索 | **大** |
| 响应速度 | <100ms 感知 | 正常（Next.js SSR） | 中 |
| 拖拽体验 | 看板拖拽 | ❌ 无拖拽 | **大** |
| 骨架屏/加载态 | 精致 | 基本（fade-rise 动画） | 中 |
| 快捷键创建 | ⌘N 任意位置创建 | 需导航到具体页面 | **大** |
| 通知推送 | 桌面通知 | 仅站内/邮件 | 中 |
| 离线和乐观更新 | ✅ | ❌ | 大 |

**追赶 Linear 的建议**：当前不追求。回路OS 的用户群（组织治理者）对实时协作和毫秒级响应的需求低于 Linear（开发者日常工具）。但 `⌘K` 的深度整合和看板拖拽值得在 v3 中投入。

### 4.3 差异化护城河评估

回顾 V1 战略报告定义的三层护城河：

| 护城河层级 | V1 状态 | V2 状态 |
|-----------|---------|---------|
| **第一层：方法论编码** | 70% | **90%** — 张力/阻塞点合并、行动归属到角色、归属确认卡、防双线守护（数据模型层） |
| **第二层：AI 教练闭环** | 0% | **75%** — 三角色代码完整，协作协议已定义，但未经真实组织数据验证 |
| **第三层：方法论验证的社会证明** | 0% | **0%** — 尚未在真实组织中运行 |

**V2 护城河现状**：第一层加固完毕，第二层实质性建立（从 0% → 75%），第三层仍是远期目标。

## 六、下一步迭代计划

### 6.1 近期 3 个高影响力低投入优化（≤2 周，≤15h）

这三项可以在正式启动 v3 前快速完成，立即提升系统可信度：

#### 优化 1：AI 翻译质量追踪（4h，最高 ROI）

**问题**：06 文档承诺「翻译准确率 ≥85%，否则降级」，但当前无任何追踪机制。无法知道 AI 表现如何。

**方案**：
- 在 Tension 表增加 `translationAccepted` 布尔字段（用户是否接受了 AI 翻译而未修改）
- 在 `createTensionAction` 中：若 `aiTranslation` 存在，比较 `description` 与 `aiTranslation` 的一致性（简单字符串相似度或新增字段标记）
- 首页增加一个简单的 AI 质量卡片：「本周 AI 翻译采纳率 X%，低于 85% 时标红」
- 4 小时足够实现整个追踪闭环

#### 优化 2：Worker 启动指南 + 健康检查（2h，消除最大运营风险）

**问题**：Worker 是独立进程，无文档说明如何启动和监控。这是当前最大的「代码完工但不可运行」风险。

**方案**：
- 在项目根目录创建 `WORKER.md`：说明 `pnpm worker` 启动方式、PM2 配置示例、Docker Compose 示例
- Worker 增加健康检查端点（health check HTTP endpoint）
- 首页概览显示「Worker 状态：运行中 / 离线」，离线时标红

#### 优化 3：修复看板「待闭环」趋势数据错误（1h，修复明显 bug）

**问题**：首页「待闭环」卡片的趋势数据用的是 `trend.tensions`（张力新增趋势），而非「待闭环阻塞点」的独立趋势。这是一个数据展示错误——用户看到的是「本周新增张力 ↑30%」但卡片标题是「待闭环」。

**方案**：
- 在 `getWeeklyTrend` 中增加本周期 vs 上周期活跃阻塞点计数
- 或在首页直接使用正确的趋势数据

### 7.2 产品化的前提条件（必须全部满足）

**硬条件（缺一不可）**：

1. ✅ **技术可行性已验证** — V2 证明回路制可以工具化
2. ❌ **治理有效性已验证** — 需要至少 1 个组织运行 1 个月，闭环率 ≥60% 或 TTR 下降 ≥20%
3. ❌ **AI 可信度可量化** — 需要 ≥1 个月的 AI 准确率追踪数据，证明 ≥85%
4. ❌ **系统稳定性可证明** — 需要 ≥99% uptime 的监控数据，至少 1 个月

**软条件（加分项）**：

5. ❌ 2-3 个回路负责人的 NPS ≥30
6. ❌ 创始人/决策者的治理日志确认率 ≥50%
7. ❌ 有结构化的产品文档和使用指南

### 7.3 产品化路线（如果验证数据正面）

    ├── 社区建设（飞书群/Slack）

### 7.4 诚实建议：不要急于产品化

V1 战略报告的核心建议是「验证比功能多更重要，闭环率比代码行数更重要」。这个建议在 V2 仍然成立，而且更加紧迫——**因为 V2 的功能已经很多了，但没有验证**。

V3 应该完成验证闭环，而不是继续堆功能。如果 v3 的验证数据显示回路制在这个组织中运转有效（TTR 显著下降），产品化就有了坚实的数据基础。如果数据显示无效，那么产品化是空中楼阁。

## 八、北极星指标追踪状态

### 8.1 GDS 四个子指标的可追踪性

回顾 V1 定义的北极星指标：**治理密度分数（GDS）**

```
GDS = (闭环率 × 0.4) + (人均周张力数 × 0.25) + (会议按时完成率 × 0.2) + (采纳健康度 × 0.15)
```

| 子指标 | V1 状态 | V2 状态 | 数据源 | 可追踪？ |
|--------|---------|---------|--------|---------|
| **闭环率** | 静态计数 | ✅ `getCloseoutRate()` | Tension 表 status=RESOLVED / 总数 | ✅ **已可追踪** |
| **人均周张力数** | 不可得 | ⚠️ 有周环比但缺分母 | Tension 表 + 活跃用户数 | 🟡 数据有，需计算 |
| **会议按时完成率** | 不可得 | ❌ 未实现 | Meeting 表 durationMin vs 上限 | ❌ 需新增 |
| **采纳健康度** | 不存在 | ✅ `getAdoptionHealth()` | 综合评分（张力活跃度+闭环率+闭环数） | ✅ **已可追踪** |

### 8.2 V2 首页已有的追踪能力

当前首页展示的健康度指标：

- **闭环率**：百分比展示，目标 ≥70%
- **周新增张力数**：本周 vs 上周趋势
- **周闭环数**：本周绝对数
- **采纳健康度评分**：0-100 分制，四级分类（dormant/waking/breathing/thriving）
- **活跃回路数**：静态计数
- **感知张力数**：当前 OPEN 状态的张力
- **待闭环数**：活跃状态的张力（含 OPEN/ASSIGNED/IN_PROGRESS/BLOCKED）

### 8.3 与 V1 定义的差距

| V1 定义的需求 | V2 实现状态 |
|-------------|-----------|
| 闭环率 7 天趋势 | ✅ 周环比趋势（↑/↓ 百分比） |
| 闭环率 30 天趋势 | ❌ 未实现（仅本周 vs 上周） |
| 人均张力数 | ⚠️ 有周总量但缺少活跃用户分母 |
| 会议按时完成率 | ❌ 完全未实现 |
| AI 翻译采纳率 | ❌ 未追踪（详见优化 1） |
| 通知响应率 | ❌ 未追踪 |
| 按回路拆解 | ❌ 未实现 |
| 反作弊护栏 | ❌ 未实现（如闭环率>90%+人均<0.5 告警） |

### 8.4 v3 北极星指标完善建议

1. **补齐会议按时完成率**：在 Meeting 创建/结束时记录，追踪 ≤30min战术会/≤90min治理会的占比
2. **增加「人均」维度**：需要活跃用户计数逻辑（过去 7 天有登录）
3. **计算完整 GDS**：首页陈列四个子指标 + 加权总分
4. **反作弊护栏**：三种告警规则化（刷闭环/张力倾泻/盲点确认）

### 9.2 V2 相比 V1 的三大质变

1. **从「AI-ready」到「AI-powered」**：不再有 stub。三角色的真实 LLM 集成、多 provider 抽象、降级策略、草稿确认机制——AI 从概念变成了可运行的代码。这是 V1→V2 最根本的变化。

2. **从「静态记录工具」到「会主动说话的系统」**：通知铃铛、邮件提醒、Worker 定时扫描、DDL 临近提醒——系统不再是等待用户来查看的死数据，而是主动 push 信息的活体。这与设计文档「活体组织美学」的隐喻真正对齐了。

3. **从「CRUD 应用」到「治理工作流引擎」**：会议中张力→行动项/决议的转换、治理日志的月度起草-发布工作流、宪章的创建-批准-归档版本链——系统开始体现出「治理」的语义，而非仅仅「数据管理」。

### 9.3 V2 的最大短板

1. **零验证**（最关键）：所有功能都在代码中，但没有在真实组织中运行过。AI 准确率是多少？通知是否扰民？闭环率有没有提升？——这些 V1 战略报告反复强调的「验证问题」，V2 仍然没有回答。

2. **零测试**：85% 的功能交付度 vs 2.5/10 的测试评分。这在内部工具阶段可以接受，但每多写一行无测试的代码，产品化的难度就增加一分。

3. **零监控**：Worker 是否在运行？AI 调用是否成功？系统有没有崩溃？——没有任何可观测性基础设施。这是一个定时炸弹。
