# 三视角审查任务说明

## 模式：模式B（批判性审查/方案验证）

三位审查者从各自独立的专业视角，对回路OS 系统设计文档进行严格审查。
三位审查者**互不通气**，独立输出审查报告。

## 审查要求（共同）

- 逐项评分（1-10分），标注风险等级（🔴高/🟡中/🟢低）
- 每个判断必须有具体证据（引用文档原文）
- 输出 8000-12000 字的独立审查报告
- 最终综合评分（加权平均）

## 角色 1：组织治理架构师
**文件**: requirements/governance-review.md

审查视角：Holacracy 组织治理理论、角色-归属模型、会议节奏设计、冲突升级机制、宪章演进

核心审查问题：
- 回路制与 Holacracy 的融合是否自洽？
- 归属 vs 角色的数据模型是否正确且可操作？
- 双层会议体系设计是否有理论依据和实践可行性？
- 冲突升级 L1-L4 的设计是否完备？
- 组织宪章/治理日志的进化闭环是否可运转？

## 角色 2：AI产品技术架构师
**文件**: architecture/tech-review.md

审查视角：系统架构、数据模型设计、AI Agent 能力、飞书原生技术可行性、可扩展性

核心审查问题：
- 四层架构是否合理？层间接口是否清晰？
- 7 张主表 + 4 张子表的数据模型是否完备？
- AI 教练三角色是否真正"AI 原生"？
- 飞书多维表格+机器人的技术限制？
- 可配置+预置的通用化程度？

## 角色 3：组织变革与商业顾问
**文件**: research/business-review.md

审查视角：组织采纳可行性、竞争差异化、市场定位、用户迁移路径、商业可持续性

核心审查问题：
- 从无到有的迁移路径是否可行？
- 与 Glassfrog 的差异化是否构成壁垒？
- "AI 原生组织行业标杆"的定位是否被设计支撑？
- 三批交付路线是否合理？
- 商业可持续性如何？

## 整合阶段

三个Agent完成后，进入整合与交叉验证：
1. 评分收敛分析
2. 逐项交叉验证矩阵
3. 隐含致命假设清单合并
4. 整合总报告
