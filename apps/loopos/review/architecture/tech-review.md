# 回路OS（LoopOS）技术架构审查报告

> **审查人**：AI产品技术架构师（委托审查）
> **审查日期**：2026-07-07
> **审查对象**：回路OS · 系统设计总览（v2，已批准）
> **审查范围**：四层架构、数据模型、AI Agent三角色、飞书平台技术方案、冲突升级自动化、可配置设计、文档完整度
> **当前状态**：零代码实现，仅设计文档层

---

## 一、执行摘要

回路OS是一个雄心勃勃的组织治理操作系统设计，其核心理念——将回路制方法论通过飞书原生工具落地、以AI Agent承担教练角色——在概念层面具有明显的创新性和差异化价值。然而，从技术架构的严谨性来看，当前设计文档存在三个结构性缺陷：

1. **层级耦合度过高，接口定义缺失**：四层架构的层级边界仅停留在概念层面，层间数据契约、API契约、事件契约均未定义。L2感知层同时承担"感知"和"执行"双重职责，破坏了单向数据流原则。
2. **飞书平台假设过于乐观**：设计中对飞书多维表格API的QPS限制（搜索20次/秒、批量500行/次）、事件回调3秒超时、群机器人无会话状态等关键约束缺乏认知，导致多个核心功能在技术上不可行的风险。
3. **AI能力假设缺乏验证路径**：教练三角色（尤其是"检测禁用词/翻译张力/起草日志"）依赖LLM在高精度场景下的可靠性，但未给出准确率目标、降级策略和人工兜底机制。

**综合评分：5.2 / 10**（概念设计 7.5，架构严谨性 4.0，技术可行性 4.5，可实施性 4.5）

## 二、逐项评分

### 2.1 四层架构设计（评分：5.5 / 10） 🔴 高风险

#### 2.1.1 层级划分合理性

**正面评价**：
- L1（状态层）→ L2（感知层）→ L3（节奏层）→ L4（进化层）的四层抽象在概念上是合理的，符合"数据→感知→执行→进化"的递进逻辑。
- 载体分工清晰：多维表格=状态、机器人=交互、文档/Wiki=记忆、日历/会议=节奏。这是设计的亮点之一。
- 与Holacracy的Circle/Role/Tension概念映射在L1层做了明确对齐。

**问题与证据**：

**问题1：层级职责溢出。** L2感知层定义包含了"张力/阻塞点、冲突升级L1-L4、教练Agent、防双线守护"四个子模块，但实际上这四个子模块的职责横跨了数据感知（读表）、规则判断（升级逻辑）、LLM推理（教练Agent）、主动推送（催闭环），本质上是一个"上帝层"。感知层应该只负责信号检测和异常告警，而冲突升级的决策逻辑和教练Agent的执行动作应该分离到独立的决策层或编排层。当前设计下，L2 = 感知 + 决策 + 执行，三合一。

> **证据**：`00-系统设计总览.md` 第六节将"冲突升级L1-L4自动化"和"教练Agent三角色"都放在L2层，但前者是规则引擎/状态机逻辑，后者是LLM驱动的人机交互，两者在技术实现上完全不同。

**问题2：数据流向不清晰。** 四层架构的ASCII图中用"←"表示L4←L3、L4←L2的关系，但未定义数据如何从L1向上流动到L2再到L3再到L4。例如：
- L2的"冲突升级"需要读取L1的哪些表？写入哪些表？
- L3的"战术会议程"是读取L1的追踪表还是L2的升级标记？
- L4的"治理日志AI草稿"聚合了哪几层的数据？

> **证据**：设计文档中只有一句"机器人背后读写表格"，但从未说明任何一条具体的数据流路径。`06-系统设计总览.md`的第八节（进化闭环）提到"运转→张力→追踪→模式→规则→运转"，这是一个业务流程闭环，不是数据流。

**问题3：层间接口完全未定义。** 没有任何接口契约（API/事件/消息格式），层与层之间是隐式耦合。举一个具体场景：L2的"阻塞点48h无更新@负责人"——这需要L1追踪表有一个"最后更新时间"字段；需要L2有一个定时轮询/事件监听机制；需要L2能读取L1的"负责人"字段并发飞书消息。这三者之间的调用链、错误处理、重试策略完全没有提及。

### 2.2 数据模型设计（评分：4.5 / 10） 🔴 高风险

#### 2.2.1 表结构完备性

**正面评价**：
- 7张主表+4张子表的结构覆盖了Holacracy的核心实体：Circle(回路)、Role(角色)、Person(人员)、Tension(张力)、Action(追踪)。
- "归属 vs 角色"的分离设计是数据模型的灵魂，解决了Holacracy中Person-Role-Circle三角关系中最容易混淆的部分。
- 回路间接口表(表6)显式建模了跨回路的契约关系，这在同类治理系统中不常见，是差异化设计。

**问题与证据**：

**问题1：7张主表没有一张有字段定义。** 设计文档只列出了表名和"行的语义"，但没有任何一个字段被定义。以下关键字段缺失导致无法评估模型完整性：

| 表名 | 必须定义但缺失的字段 |
|------|---------------------|
| 回路 | 回路ID、名称、目的/使命、负责人ID、父回路ID（嵌套回路）、状态（活跃/归档）、创建时间 |
| 角色 | 角色ID、名称、目的、所属回路ID、承担人ID列表、角色类型（主归属/跨回路支援）、契约关联接口ID |
| 人员 | 人员ID、飞书user_id、主归属回路ID、归属确认卡状态（未签/已签/待更新）、入职日期 |
| 张力·阻塞点 | 张力ID、描述、提出人、提出时间、严重等级(L1-L4)、当前状态、所属回路ID、关联角色ID、关联接口ID |
| 追踪表 | 阻塞点ID、负责人、承诺闭环日期、实际闭环日期、上次更新、升级次数、当前L层级 |
| 回路间接口 | 接口ID、提供方回路ID、消费方回路ID、交付物描述、SLA、状态、关联角色ID列表 |
| 指标·里程碑 | 指标ID、回路ID、Phase、目标值、实际值、测量日期、指标类型（领先/滞后） |

> **证据**：设计文档标明"字段详设见 `01-数据模型与表结构.md`"，但该文件在仓库中不存在（参见第七节文档完整度审查）。

**问题2：张力与阻塞点的概念混用。** "张力·阻塞点"表（表4）将两个不同语义的概念放在同一张表中，但没有区分字段。在Holacracy中：
- **张力（Tension）** = 感知到的"实际状态与理想状态之间的差距"，是一种信号，不一定可行动。
- **阻塞点（Blockage）** = 已被确认为需要闭环的事项，有明确的负责人和截止日期。

当前设计将两者放在一张表，可能导致：无法区分"有人觉得不对劲"和"这件事需要做"；追踪表（表5）与张力表之间的一对多/一对一关系不明确。

**问题3："追踪表"与"张力·阻塞点"表的关系不清晰。** 追踪表的行语义是"一个待闭环阻塞点"，张力表的行语义是"一个张力或阻塞点"。两者之间的数据流转规则未定义——阻塞点何时从张力表迁移到追踪表？还是追踪表是张力表的一个视图/状态子集？

**问题4：4张子表的定义更加模糊。** "归属确认卡、会议纪要、治理日志、组织宪章"被列为子表，但它们更可能是：
- 归属确认卡 → 人员表的一个字段/关联记录
- 会议纪要 → 一张独立表，需要字段如：会议ID、日期、类型(战术会/治理会)、所属回路、议程、决议、参与者
- 治理日志 → 一张独立表，或Wiki文档的索引表
- 组织宪章 → Wiki节点，不适合用多维表格承载

**问题5：缺失关键实体。** 以下实体在治理系统中至关重要但未被建模：
- **决策记录**：谁在什么会上做了什么决策，依据是什么（治理会产生的决议需要有独立记录）
- **议程项**：治理会/战术会的议程条目，需要与张力/阻塞点关联
- **历史快照**：回路结构变更、角色变更、归属变更的时间线（没有审计日志就无法追溯治理演进）
- **通知/提醒记录**：系统发出的提醒、@消息、升级通知需要记录，否则无法追踪"48h无更新"的触发历史

#### 2.2.2 表间关联关系

设计中提到了关联（link）的概念（如"角色表里跨回路支援角色没有 link 到回路间接口契约则自动标红"），但以下关键关联完全未定义：

- 人员 ↔ 回路：通过主归属字段关联，但1人1归属的约束如何实现？唯一索引？
- 角色 ↔ 回路：角色属于哪个回路？角色可以有多个回路吗？
- 张力 ↔ 回路/角色/接口：一个张力可能涉及多个回路、多个角色，关联字段是单向还是双向？
- 追踪 ↔ 张力：是外键引用还是独立记录？

### 2.3 AI教练三角色设计（评分：4.0 / 10） 🔴 高风险

#### 2.3.1 "AI原生"程度评估

**正面评价**：
- "AI做教练，不做决策者"的定位原则是正确的，避免了AI越权的伦理风险。
- 三角色分工（会议守护者/张力翻译者/机制建设者）映射了人类教练的三项核心职能，分工逻辑清晰。
- "不碰权力只建系统"的硬约束体现了对AI能力边界的基本敬畏。

**问题与证据**：

**问题1：角色边界模糊——会议守护者到底做什么？**

"检测禁用词（可能/大概/争取）、阻塞点四要素齐全、超时"——这三个任务的难度差异极大：
- **检测禁用词**：关键词匹配即可，不需要AI。用正则或敏感词表可以做到100%准确率。
- **阻塞点四要素齐全**：需要NLP理解会议发言是否包含"谁、做什么、何时完成、如何验证"四个要素。这是典型的信息抽取任务，LLM有一定的抽取能力，但准确率在真实会议场景下可能不足80%（含糊表达、省略主语、跨句指代都是挑战）。
- **超时检测**：纯程序化计时，不需要AI。

当前设计将三个完全不同技术难度的任务打包在同一个角色名下，但没有任何技术方案说明如何实现。更严重的是：会议守护者需要在"会议进行中"实时工作，这要求它实时监听会议发言并即时反馈——飞书机器人有实时语音转文字API吗？如果依赖人工输入会议纪要，那"实时守护"就无从谈起。

> **关键问题**：飞书机器人无法自动接入视频会议中的语音流。会议守护者要么依赖会后纪要分析（此时已失去"守护"的实时性），要么依赖参会者手动@机器人（破坏了会议节奏）。这是一个**隐含致命假设**（见第四节）。

**问题2：张力翻译者——翻译的准确率没有目标**

"模糊吐槽→结构化张力"是一个典型的NLU（自然语言理解）任务。在真实的组织场景中，"吐槽"的表达方式极其多样：
- "数据团队又卡我们了" → 需要LLM推断出：谁卡了谁？什么被卡了？阻塞点在哪？
- "这个需求做了三周还没排上" → 需要推断：哪个需求？归哪个回路？谁应该负责？
- "XX领导说要先看" → 需要识别双线汇报模式

LLM在零样本场景下做信息抽取的准确率通常在70%-85%之间（参考最新的信息抽取基准测试如GLiNER、NuNER等）。对于治理系统来说，如果翻译错误率达到15%-30%，意味着每3-7条张力就有一条被错误归类、错误指派，系统会产生大量噪音，反而增加治理成本。

**问题3：机制建设者——AI起草日志的质量和风险**

"月初从追踪表起草治理日志"——这个任务在技术上是可行的（LLM擅长摘要生成），但质量取决于追踪表数据的结构化程度。如果追踪表的阻塞点描述含糊（"推进XX事项"），LLM生成的日志将毫无价值。

"季度起草宪章变更"——这是一个极高风险的任务。宪章是组织的治理宪法，涉及权力分配、决策权限、资源归属等敏感事项。AI草稿的错误（如误删除某回路的决策权）可能导致组织混乱。当前设计没有给出AI草稿的人工审核流程、版本对比机制、和回滚策略。

**问题4：三个Agent角色之间如何协作？**

设计文档将三角色并列，但未定义它们的协作关系：
- 张力翻译者翻译出来的结构化张力，是否自动交给会议守护者作为议程输入？
- 会议守护者在会上检测到新的阻塞点，是否触发张力翻译者对会议纪要的二次处理？
- 机制建设者的日志草案是否需要张力翻译者预处理追踪表数据？

这个协作关系的缺失意味着三角色在实际实现时可能是三个独立的prompt，彼此之间数据不互通，形成三个信息孤岛。

### 2.4 飞书多维表格+机器人技术方案（评分：4.5 / 10） 🔴 高风险

#### 2.4.1 多维表格API限制分析

经过对飞书开放平台文档的调研，以下已知限制直接影响回路OS的设计可行性：

| 限制项 | 具体数值 | 对回路OS的影响 |
|--------|---------|---------------|
| 单次查询最大行数 | 500行 | 查询"哪些事卡了两周以上"时，如果追踪表超过500行需分页，分页逻辑未设计 |
| 搜索API频率限制 | 20次/秒 | 多个用户同时查询时可能触发限流 |
| 批量创建最大行数 | 1000条/次 | L3会议纪要自动回写可能一次需要更新多条记录 |
| 单表最大列数 | 300列 | 7张主表如果每张表字段较多，部分表可能接近上限 |
| 公式+查找引用字段 | 100个/表 | 如果大量使用公式字段做状态计算，可能受限 |
| 单表附件上限 | 20,000个 | 会议纪要附件、归属确认卡扫描件可能累积 |
| 行数上限 | 免费版1000行/表，商业版数万行，企业版可扩容至200万 | 张力记录随时间增长，免费版/商业版可能在数月到一年内触顶 |
| 并发编辑 | 200人 | 全体成员同时在线编辑不会超（中小组织），但需注意 |
| 评论总数 | 20,000条/Base | 如果在追踪表中用评论做讨论，可能触顶 |

**关键风险**：设计中提到的"机器人背后读写表格"过于笼统。具体来说：
- **查询类指令**（"我欠什么""回路二本周怎么样"）：需要跨表关联查询（人员表→追踪表→回路表），但多维表格不支持SQL JOIN，只能通过API逐表查询后在应用层组装数据。这意味着一次查询可能触发3-5次API调用，在20QPS限制下，并发用户数>4就会触发限流。
- **写入类操作**（提张力→写入张力表）：每次@机器人提张力，触发1次写入API。在团队规模30-50人时问题不大，但需要监控。

#### 2.4.2 机器人交互的并发与状态管理

**问题1：飞书事件回调3秒超时。** 飞书开放平台要求事件回调在3秒内返回HTTP 200。但回路OS的核心交互都涉及LLM调用：
- "提张力→结构化翻译→写入表格"：LLM推理+API调用+表格写入，总耗时通常在2-8秒。
- "我欠什么→查询多张表→LLM组织语言→回复"：3次+API调用+LLM推理，耗时5-15秒。

这意味着回路OS的机器人后端必须采用异步架构：收到消息后立即返回200，将消息放入队列，后台处理完成后通过"发送消息API"主动推送给用户。当前设计文档完全没有提及这一架构需求。

> **证据**：飞书开放平台文档明确说明"业务handler处理耗时超过3s，触发消息重推"。("常见问题-服务端API"中的事件FAQ)

**问题2：群聊中的会话状态管理。** 设计文档暗示机器人可以在群聊中工作（"战术会前自动发议程卡"显然是群消息），但飞书群机器人的消息事件中，每次消息是独立的，没有"会话"概念。这意味着：
- 用户说"提张力 数据团队卡了我们"，机器人需要额外推断上下文（哪个回路？哪个接口？）
- 如果用户分两条消息说"提张力"和具体描述，第二条消息无法关联到第一条
- 多轮对话澄清（"请问是哪个回路的阻塞？"→用户回复）需要自行维护会话状态

当前设计没有考虑会话状态管理方案（Redis/数据库存储上下文）。

**问题3：消息去重。** 由于3秒超时可能导致飞书重推消息，如果后端没有幂等处理，同一条"提张力"指令可能被处理两次，创建两条重复的张力记录。设计未提及去重策略。

#### 2.4.3 日历/会议集成

设计中提到"日历/会议=节奏"，暗示飞书日历API集成。但飞书日历API有以下限制：
- 需要额外的权限scope
- 创建/修改日历事件需要组织成员授权
- 自动创建周期性会议（每回路每周2-3次战术会）需要处理时区、节假日、成员变更
- 会议纪要的自动关联需要会议ID映射

这些集成的技术复杂度在当前设计中完全没有体现。

### 2.5 冲突升级L1-L4自动化（评分：3.5 / 10） 🔴 高风险

#### 2.5.1 状态机缺失分析

设计文档对冲突升级的描述仅有三行：

```
- L1（回路内）→ 战术会上当场到人
- L2（回路间接口延迟）→ 次日提醒+超时升L3
- L3（不可调和）→ 自动加入治理会议程标"需决策者拍板"
- L4（战略级）→ 归档战略回路
```

这四行描述了四个升级目标，但完全没有定义升级的条件判断逻辑。一个可工作的状态机至少需要：

**缺失1：状态定义。** 一个阻塞点/张力有哪些可能的状态？至少需要：
- `OPEN`（已提出）
- `ASSIGNED`（已指派）
- `IN_PROGRESS`（处理中）
- `BLOCKED`（被阻塞）
- `ESCALATED_L2/L3/L4`（已升级）
- `RESOLVED`（已闭环）
- `REJECTED`（被驳回/降级）

当前设计未定义任何一个状态枚举。

**缺失2：状态转移条件。** 什么条件触发了从L1到L2的升级？"回路间接口延迟"——延迟的定义是什么？超过SLA 1天？3天？SLA从哪里读取（接口表有SLA字段吗）？如果接口表根本没有SLA字段，延迟就无法判断。

**缺失3：升级的数据依赖。** 判断"不可调和"需要什么数据？
- 该阻塞点在追踪表中的反复出现次数？
- 关联的回路间接口是否有多个冲突契约？
- 决策者是否已经参与过讨论但无法达成一致？
这些数据在当前7张主表+4张子表中，哪些表承载？如果不承载，升级判断就依赖AI的"主观判断"——但AI不可以做决策者（是硬约束），那谁来定义"不可调和"？

**缺失4：降级机制。** 只有升级路径，没有降级路径。一个被升级到L3的阻塞点，如果回路间自行协商解决了，如何从L3降回L2？如果缺乏降级机制，L3会不断堆积已解决的问题，使得治理会失去焦点。

**缺失5：冲突的来源分类。** 设计提到"冲突升级"但没有区分冲突类型：
- 资源冲突（两个回路抢同一个人的时间）
- 优先级冲突（回路A的P0 vs 回路B的P1谁先）
- 语义冲突（对同一需求的理解不同）
- 权力冲突（谁有权决定某个事项）
不同类型的冲突需要不同的升级路径和裁决逻辑。当前设计一刀切。

#### 2.5.2 自动化升级的可行性

"冲突升级L1-L4自动化"是一个美好的愿景，但在当前数据模型下，真正可自动化的是：
- **定时器触发**：阻塞点超过N天未闭环→自动升级（最容易实现）
- **规则触发**：接口契约状态异常→自动升级（需要接口表有状态字段）
- **不可自动化**："不可调和"的判断需要人类判断，AI最多只能提示风险信号

建议将"自动化"的范围收窄为"自动检测升级信号+半自动推荐升级"，由人类确认是否真正升级。

#### 2.5.3 评分明细

| 评估维度 | 评分 | 说明 |
|---------|------|------|
| 状态枚举 | 1/10 | 零状态定义 |
| 转移条件 | 1/10 | "延迟""不可调和"无量化标准 |
| 数据依赖 | 2/10 | 升级判断所需的字段不明确 |
| 降级路径 | 0/10 | 缺失 |
| 自动化范围 | 5/10 | 定时期限升级可实现，语义判断不可自动化 |
| **综合** | **3.5/10** | |

### 2.6 "可配置+预置"设计（评分：5.0 / 10） 🟡 中风险

#### 2.6.1 通用化程度评估

**正面评价**：
- 示例预置包的思路正确：先让用户用起来，再逐步暴露配置能力。
- 区分了"回路数/名称/目的、指标体系、会议节奏、角色/张力/升级规则语义"这几个可配置维度，方向正确。

**问题与证据**：

**问题1：配置的声明能力边界未定义。**

"可配置"是一个光谱，从"改几个参数"到"完全自定义"差异巨大。当前设计需要明确回答：
- 新增一个回路类型需要改什么？是改JSON配置文件？还是改多维表格的行数据？还是改代码？
- 指标体系：阶梯指标是固定三阶还是可配置为N阶？指标的聚合方式（平均/加权/最小值）是否可配置？
- 会议节奏：如果某个回路只需要每周1次战术会而非2-3次，如何配置？
- 禁用词列表：是可编辑的单词表还是需要改prompt？

如果没有明确的配置schema（JSON Schema或类似声明），"可配置"就只是一个营销词汇。

**问题2：预置包的真实依赖。**

设计提到预置包源自"某大模型团队的真实生产链路"，这意味着预置的回路结构（战略零/数据/预训练/后训练/工程基座）是AI研发团队的特化配置。如果另一个行业（如电商、金融、制造业）的组织想用回路OS：
- 5回路模板是否适用？制造业的生产链路是"设计→采购→生产→质检→物流"，需要5个不同的回路。
- 阶梯指标（1月闭环率50%/3月核心指标进前5/6月综合前7/12月前5）显然是AI模型评估的指标，其他行业的阶梯指标完全不同。
- 角色的语义（"预训练负责人"）在其他行业不存在。

这意味着当前的"可配置+预置"实际上是"预置为主，配置为辅"，通用化程度远低于设计文档声称的水平。真正的通用化需要的是一个**回路工厂（Circle Factory）**——允许用户定义回路的目的、成员、角色模板、指标体系，系统自动建表。

**问题3：配置的版本化和迁移。**

如果用户使用预置包上线后，修改了大量配置（新增回路、调整指标、改变升级规则），当预置包有更新时（如修复了某个Agent prompt的bug），如何合并更新而不覆盖用户的配置？这是一个典型的"fork vs merge"问题，当前设计完全没有涉及。

#### 2.6.2 新增回路类型的影响面分析

假设用户想新增一个"安全合规回路"，需要修改：
1. **回路表**：新增一行 ✅ 简单
2. **角色表**：新增安全合规相关的角色定义（安全负责人、合规审查员等）
3. **人员表**：重新分配部分人员的归属 ⚠️ 可能触发归属确认卡重签
4. **回路间接口表**：定义新回路与现有5个回路之间的接口契约
5. **指标表**：定义安全合规指标的阶梯目标
6. **教练Agent prompt**：安全回路的会议是否需要特殊的禁用词/阻塞点四要素？⚠️ 可能需要修改prompt
7. **自动化规则**：新回路的阻塞点升级阈值是否与其他回路相同？

粗略估计，新增一个回路类型涉及7张主表中的6张+可能的prompt修改。这不是"改一行配置"能完成的。

## 三、结构性缺陷与盲区

### 3.1 缺陷1：缺少中间编排层（P0级）

当前四层架构中，L1/L2/L3/L4之间的所有交互都隐性依赖机器人这一单一入口。但实际上需要一个**编排层（Orchestration Layer）**来管理：
- 定时任务（每48小时检查未闭环阻塞点、每月起草治理日志、每季度起草宪章变更）
- 事件驱动（人员归属变更→触发三道防线检查、张力状态变更→触发升级判断）
- 工作流引擎（会议议程生成→纪要回写→追踪表更新→通知发送的串联）

建议在L2和L3之间新增一个**L2.5 编排层（Workflow Engine）**，或在L2内部明确分离"感知"和"编排"的职责。

### 3.2 缺陷2：缺少通知/告警子系统（P0级）

设计中多处提到"提醒""通知""@负责人"，但没有定义统一的通知机制：
- 通知的渠道是什么？飞书消息卡片？群消息？私聊消息？
- 通知的频率控制如何避免"通知轰炸"？
- 通知的确认/已读/忽略状态如何追踪？

### 3.3 缺陷3：缺少权限与访问控制模型（P1级）

一个治理系统必然涉及敏感数据（归属关系、角色分配、绩效指标）。当前设计没有定义：
- 谁可以查看所有回路的数据？（决策者？全员透明？）
- 谁可以修改归属关系？（仅决策者？回路负责人？）
- 谁可以读写张力表？（提张力的人可以编辑自己的张力吗？）
- AI Agent的读写权限边界在哪？

飞书多维表格支持高级权限设置（按字段级别的权限控制），但当前设计完全没有利用或描述权限方案。

### 3.4 缺陷4：缺少错误处理与降级策略（P1级）

当系统依赖的外部服务不可用时（LLM API故障、飞书API限流、多维表格行数触顶），回路OS应该如何降级运行？例如：
- LLM不可用时，张力捕获是否回退到纯表单填写？
- 多维表格行数触顶时，是否自动归档旧数据？
- 机器人3秒超时时，用户看到的是即时确认还是5秒后的延迟回复？

### 3.5 缺陷5：缺少数据归档与生命周期管理（P1级）

张力、阻塞点、会议纪要会随时间累积。设计文档提到了"组织记忆"但未定义数据的生命周期：
- 已闭环的阻塞点保留多久？
- 历史数据进行归档还是直接删除？
- 归档后是否影响AI的治理日志起草（月度日志需要至少当月的完整数据）？

### 3.6 缺陷6：缺少可观测性设计（P2级）

作为组织治理系统，回路OS的运行质量本身就是治理对象：
- 系统本身的关键指标是什么？（机器人响应延迟、张力翻译准确率、闭环率趋势、API错误率）
- 数据异常如何监控？（如某个回路连续一周没有新张力——是运转良好还是成员不再使用？）
- 错误日志和审计追踪如何实现？

                    └── DEESCALATED ←──────┘
```

每个状态转移需定义：
- 触发条件（量化阈值，如"超过48小时未更新"）
- 所需数据字段（明确读取哪张表的哪个字段）
- 转移后的动作（通知谁、写什么表、发什么消息）

**P0-4：设计异步消息处理架构**

明确机器人后端的架构选型：
- 事件接收层：立即返回200，消息入队（Redis/RabbitMQ/飞书消息队列）
- 工作处理层：消费队列，调用LLM+API，完成后通过飞书消息API回复
- 幂等处理：基于event_id/message_id做去重
- 会话状态管理：多轮澄清对话的状态存储方案

**P0-5：定义AI能力的量化目标和降级策略**

为每个AI功能设置明确的准确率目标和降级方案：

| AI功能 | 最低可接受准确率 | 测量方法 | 降级策略 |
|--------|---------------|---------|---------|
| 张力翻译 | ≥85% | 每周人工抽检20条 | 低于85%时关闭自动翻译，改为表单引导模式 |
| 双线汇报识别 | ≥70%召回 | 治理会上对照人工发现 | 仅做告警标记不自动升级 |
| 日志起草 | ≥90%关键事件覆盖 | 回路负责人复核 | 人工补充再发布 |
| 宪章变更 | 不允许独立发布 | 100%人工审核 | 变更草案+diff视图+审核工作流 |

- 通知频控策略（同一阻塞点24小时内最多2次提醒）
- 通知确认追踪（已读/已处理状态）

**P1-3：定义权限模型**

分层定义：
- **角色级权限**：回路负责人、回路成员、决策者、系统管理员
- **表级权限**：哪些角色可读写哪些表
- **字段级权限**：敏感字段（如绩效指标）的可见性控制
- **Agent权限**：明确AI只能写入哪些字段（如：不能改归属关系，不能改指标目标值）

**P1-4：设计配置Schema和回路工厂**

        "purpose": {"type": "string"},
        "lead_role_id": {"type": "string"},
        "tactical_meetings_per_week": {"type": "integer", "minimum": 1, "maximum": 5},
        "escalation_threshold_hours": {"type": "integer", "default": 48}
      }
    }
  }
}
```

**P1-5：数据归档策略**

定义：
- 已闭环阻塞点：保留6个月后归档到历史表
- 张力记录：保留3个月后归档
- 会议纪要：保留12个月
- 归档数据的可查询性（是否需要单独的归档Base）
