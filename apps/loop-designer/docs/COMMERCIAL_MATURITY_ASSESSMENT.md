## 二、高危问题清单（上线前必须修复）

### 🔴 严重 (P0) — 直接阻断商业化

#### 2.1 TypeScript 构建时跳过类型检查

**位置**：`next.config.ts` 第 11 行

```typescript
typescript: { ignoreBuildErrors: true }
```

**风险**：生产构建静默跳过所有 TS 类型错误，运行时错误直接暴露给用户。这是将"编译期安全网"完全拆除的行为。

**建议**：改为 `false`，或在 CI 流水线中增加 `tsc --noEmit` 作为独立检查步骤。

---

#### 2.2 无登录/注册速率限制 — 暴力破解与批量注册敞口

**位置**：
- `src/app/api/auth/email/login/route.ts` — 邮箱登录
- `src/app/api/auth/email/signup/route.ts` — 邮箱注册

**风险**：
- 登录接口无任何速率限制，攻击者可无限制尝试密码（暴力破解）
- 注册接口无任何频率控制，可被脚本批量创建账号

**建议**：
- 引入 `express-rate-limit` 或 Redis 令牌桶
- 登录：每 IP 每分钟最多 5 次尝试，连续失败 5 次锁定 15 分钟
- 注册：每 IP 每小时最多 3 个账号

---

#### 2.3 登出接口无 CSRF 保护

**位置**：`src/app/api/auth/logout/route.ts`

**风险**：恶意网站可通过嵌入 `<form action="https://loop.csi-org.com/api/auth/logout" method="POST">` 强制用户登出，属于拒绝服务攻击。

**建议**：
- 使用 Next.js Server Actions 内置的 CSRF 保护
- 或在 API 路由中增加自定义 CSRF token 校验（通过 cookie + header 双重提交模式）

---

#### 2.4 AI 模型 API Key 安全暴露在 Bearer Token 中

**位置**：`src/lib/model.ts` 第 241 行

```typescript
headers: { authorization: `Bearer ${apiKey}` }
```

**风险**：`MODEL_API_KEY` 通过环境变量注入，若日志系统未脱敏，API Key 可能出现在错误日志、请求日志中。

**建议**：
- 确认 PM2 日志和 Supabase 审计日志中不记录请求头
- 增加 HTTP 中间件在日志输出前脱敏 `Authorization` 头
- API Key 走 Secret Manager（如 Supabase Vault）而非纯环境变量

---

### 🟠 高危 (P1) — 影响数据一致性和业务安全

#### 2.5 多步数据库操作无事务保障

**位置**：
- `src/app/api/auth/join-enterprise/[code]/route.ts` — 加入企业流程（成员创建、席位递增、企业更新、邀请码消费四步非原子）
- `src/lib/app-session.ts` — `createAppSession` 中企业激活、用户更新、席位更新、会话创建四步非原子

**风险**：任一步骤失败都会导致部分写入成功、部分失败，形成脏数据。例如：
- 邀请码被消费但成员未成功创建 → 邀请码永久丢失
- 席位已递增但会话创建失败 → 席位计数虚高

**建议**：
- 使用 Supabase 的 `rpc` + PL/pgSQL 事务函数将多步操作封装为原子执行
- 或在应用层引入 Saga 模式（补偿事务）

---

#### 2.6 会话更新缺少 user_id 授权过滤

**位置**：`src/lib/sessions.ts` 第 165 行 `updateSession` 函数

```typescript
// 当前实现：仅按 enterprise_id 过滤
.eq("enterprise_id", user.enterpriseId)
// 缺少：.eq("user_id", user.id)
```

**风险**：同一企业内 A 用户可以修改 B 用户的会话数据。

**建议**：在 `updateSession` 中增加 `user_id` 过滤条件。

---

#### 2.7 密码策略过于薄弱

**位置**：`src/app/api/auth/email/signup/route.ts`

**当前**：密码仅要求 >= 6 字符，bcrypt 轮次 = 10

**风险**：6 字符无复杂度要求的密码极易被字典攻击破解。

**建议**：
- 最低 8 字符，至少包含字母 + 数字
- bcrypt 轮次从 10 提升到 12（安全性与性能的平衡点）
- 引入常见密码黑名单（如 `HaveIBeenPwned` API）

## 三、中危问题（上线后需尽快修复）

### 3.1 开放重定向风险

**位置**：`src/lib/auth-crypto.ts` 第 89-100 行 `normalizeReturnPath`

**风险**：仅检查路径前缀 `/loop-designer` 和排除 `//`，恶意构造如 `/loop-designer@evil.com` 或 `/loop-designer.evil.com/` 可能绕过检查。

**建议**：使用 `URL` 构造函数解析并强制限制为相对路径，或使用白名单机制。

### 3.3 订阅套餐席位超卖风险

**位置**：`src/app/api/admin/subscription/route.ts` 的 PATCH 处理

**风险**：降级套餐时未校验 `newSeatLimit >= used_seats`，可能导致已占用席位超过新套餐上限的数据矛盾。

**建议**：在降级前增加此校验，若 `newSeatLimit < used_seats` 则拒绝操作并提示管理员先移除多余成员。

### 3.4 飞书 OAuth 白名单默认全放行

**位置**：`src/lib/feishu-auth.ts` 第 104 行

```typescript
if (!allowedTenantKey) return; // Phase 1: 未配置时允许所有租户
```

**风险**：若忘记配置 `FEISHU_ALLOWED_TENANT_KEY`，任意飞书企业用户均可登录。生产环境必须反转此逻辑。

**建议**：生产环境改为默认拒绝，仅白名单中配置的企业可登录。

---

### 3.5 错误信息泄漏内部细节

**位置**：多个 API 路由中 `error.message` 直接返回给客户端

**风险**：异常堆栈、数据库错误、文件路径等敏感信息可能通过错误消息泄漏。

**建议**：统一错误处理中间件，在返回客户端前脱敏错误消息，改用通用提示。

## 四、架构设计评价

### 4.1 优点

- **分层清晰**：`src/lib/` (业务逻辑) → `src/app/api/` (路由层) → `src/components/` (展示层)，关注点分离到位
- **`server-only` 保护**：所有数据操作库文件正确使用了 `import "server-only"` 指令
- **Zod 校验链**：AI 模型输出经过 Zod schema 严格校验，不会将不可信数据直接入库
- **OAuth/PKCE 流程规范**：飞书 OAuth 流程使用了 PKCE，state 参数经过 HMAC 签名
- **Token 哈希存储**：认证 token 以 SHA-256 哈希存储，不存明文

### 4.2 改进空间

- **缺少 Service 层抽象**：部分 API 路由中包含大量业务逻辑（如 `join-enterprise`），应抽取到独立的 service 文件
- **错误处理策略不统一**：有的抛异常、有的返回 null、有的返回 error 对象，调用方需了解每种函数的错误约定
- **硬编码限制较多**：回答长度 6000、精炼轮次 3、会话列表 8 条等限制散落在各处，应集中管理
- **无依赖注入 / 无测试替身**：`supabase.ts` 基于环境变量直接创建客户端，无法在测试中 mock

## 九、总结

碳硅回路设计师在**功能层面**已经相对完整——5 步对话流程、AI 方案生成、多格式导出、企业管理后台均在代码层面实现。但在**商业化交付标准**上，目前有 7 项 P0 级高危问题需要在正式面向用户之前修复。

核心风险集中在三个方向：
1. **安全底线**：无速率限制 + 无 CSRF 保护 + 薄弱的密码策略 = 攻击面过大
2. **数据一致**：非事务性数据库操作可能导致不可逆的脏数据
3. **可观测性**：缺乏监控手段意味着生产故障的发现和定位全凭运气

建议按上述路线图分三阶段推进，在第一阶段的安全加固完成后即可进行小规模灰度测试，后续阶段在正式推广前补齐。
