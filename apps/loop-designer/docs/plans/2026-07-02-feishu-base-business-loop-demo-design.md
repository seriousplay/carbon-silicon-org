# 飞书多维表格业务回路示范案例设计稿

## 1. 目标

基于《回路治理：AI时代，组织如何从“管人”升级为“治理混合团队”？》中的客户投诉处理场景，设计一个可在飞书多维表格中演示的业务回路样板。

这个样板不追求覆盖完整客服系统。它只验证一件事：

> 飞书多维表格 + 工作流 + AI Agent 节点，能否把一条传统工单流程改造成一条可感知、可裁决、可验证、可回灌的业务回路。

## 2. 成功标准

演示结束时，观众应能看见四个变化：

1. 旧流程里的三类断点被显性标出：信息衰减、等待黑洞、校验真空。
2. 客户原始投诉没有被人工摘要压扁，而是被 AI 提取成结构化字段。
3. 人不再逐条搬运信息，只在低置信度、高风险或赔付承诺节点做裁决。
4. 处理结束后有回访结果和复现率回灌，下一轮分类规则可以被更新。

## 3. 设计假设

- 示范场景选用“客户投诉处理”，因为附件中已经用它解释五层系统架构，观众也容易理解。
- 首版只演示两条回路：回路 A“分类与归因”，回路 B“修复与验证”。
- 不接入真实电话系统、CRM 或客服工单系统，投诉输入先通过表单或手工新增记录模拟。
- 不处理真实客户隐私数据，演示数据全部使用虚构样例。
- AI Agent 的输出必须留有证据字段，不能只给结论。

## 4. 三种方案

### 方案 A：纯文档演示

用飞书文档或幻灯片讲清楚结构，成本最低，但观众看不到回路运行，只能理解概念。

适合内部讲解，不适合做“业务回路智能体”样板。

### 方案 B：多维表格轻量样板（推荐）

用 Base 表、视图、工作流、AI Agent 节点搭出可点击、可流转、可看板化的示范。

优点是足够轻，能在现场 10 分钟讲完，也符合附件里“明天就能在飞书多维表格上跑通第一条业务回路”的表达。

### 方案 C：Loop Designer + 飞书 Base 双系统联动

Loop Designer 负责生成回路方案，飞书 Base 负责承载运行记录。

长期价值最高，但首版演示链路偏长，也会把注意力从“回路示范”转移到系统集成。

首版建议先做方案 B，跑通后再决定是否接入 Loop Designer。

## 5. 示范叙事

旧流程：

客户投诉 -> 客服记录 -> 主管筛选 -> 经理指派 -> 处理 -> 回访

三类断点：

| 断点 | 在旧流程中的表现 | 回路化处理 |
|---|---|---|
| 信息衰减 | 客服把客户原话摘要成几句话，情绪、证据、历史上下文丢失 | 保留原始文本，AI 提取结构化字段，并附证据 |
| 等待黑洞 | 主管筛选、经理指派都在排队等人 | AI 先分类和建议分派，人只裁决异常 |
| 校验真空 | 工单关闭后没人验证同类投诉是否减少 | 回访结果、复现率、规则更新进入下一轮 |

新回路：

回路 A：投诉分类与归因

客户投诉进入 Base -> AI 提取字段 -> AI 推荐分类、根因和动作 -> 人工裁决 -> 确认后推送到回路 B

回路 B：修复与验证

责任人处理 -> 记录处理结果 -> 7 天回访 -> 统计同类投诉复现 -> 更新归因规则或异常库

## 6. Base 结构

### 表 1：投诉原始记录

用途：承接所有输入，保留原始语境。

| 字段 | 类型 | 说明 |
|---|---|---|
| 投诉编号 | 自动编号/文本 | 如 CPL-0001 |
| 投诉时间 | 日期时间 | 客户投诉进入时间 |
| 客户名称 | 文本 | 演示可用匿名客户 |
| 来源渠道 | 单选 | 电话、企微、表单、门店、邮件 |
| 原始投诉内容 | 多行文本 | 客户原话或转写文本 |
| 附件证据 | 附件 | 截图、录音、订单凭证 |
| 涉及产品/服务 | 单选/文本 | 产品线或服务模块 |
| 客户情绪强度 | 单选 | 低、中、高 |
| 是否重复投诉 | 复选框 | 由 AI 或人工判断 |
| 当前状态 | 单选 | 待结构化、待归因、待裁决、已推送修复、已关闭 |
| AI 提取摘要 | 多行文本 | 不替代原文，只作为快速阅读 |
| 关联归因建议 | 关联记录 | 关联到“AI 归因建议” |

关键视图：

- 今日新投诉：筛选“投诉时间=今天”。
- 待结构化：筛选“当前状态=待结构化”。
- 高情绪投诉：筛选“客户情绪强度=高”。

### 表 2：AI 归因建议

用途：让 AI Agent 的判断可审计。

| 字段 | 类型 | 说明 |
|---|---|---|
| 建议编号 | 自动编号/文本 | 如 CAUSE-0001 |
| 关联投诉 | 关联记录 | 对应“投诉原始记录” |
| 投诉类型 | 单选 | 交付延迟、质量问题、价格争议、服务态度、系统故障、其他 |
| 根因假设 | 多行文本 | AI 对原因的判断 |
| 建议处理动作 | 多行文本 | 推荐下一步动作 |
| 历史相似案例 | 多行文本/关联记录 | 至少给出一条证据或说明没有找到 |
| 置信度 | 数字 | 0-100 |
| 风险等级 | 单选 | 低、中、高 |
| 需要人工裁决 | 复选框 | 低置信度或高风险自动勾选 |
| AI 不确定点 | 多行文本 | 缺少哪些信息 |
| 生成时间 | 日期时间 | 建议生成时间 |

关键规则：

- 置信度低于 80，必须进入人工裁决。
- 风险等级为高，必须进入人工裁决。
- 没有历史证据时，不允许自动推送到修复回路。

### 表 3：人工裁决

用途：体现“人退到边界”，不是让人继续做信息搬运。

| 字段 | 类型 | 说明 |
|---|---|---|
| 裁决编号 | 自动编号/文本 | 如 DEC-0001 |
| 关联建议 | 关联记录 | 对应“AI 归因建议” |
| 裁决人 | 人员 | 主管或回路负责人 |
| 裁决动作 | 单选 | 确认 AI 建议、改分类、改责任人、要求补充信息、升级处理 |
| 最终投诉类型 | 单选 | 与投诉类型保持一致 |
| 最终责任团队 | 单选/人员 | 客服、产品、供应链、门店、技术 |
| 人工裁决理由 | 多行文本 | 为什么改或为什么确认 |
| 是否允许自动推送 | 复选框 | 勾选后进入修复回路 |
| 裁决时间 | 日期时间 | 人工确认时间 |

关键视图：

- 待裁决：需要人工裁决=true 且未裁决。
- 高风险裁决：风险等级=高。
- AI 被改判记录：裁决动作不等于“确认 AI 建议”。

### 表 4：修复与验证

用途：把处理结果接回下一轮，避免工单关闭后失去反馈。

| 字段 | 类型 | 说明 |
|---|---|---|
| 修复编号 | 自动编号/文本 | 如 FIX-0001 |
| 关联投诉 | 关联记录 | 对应原始投诉 |
| 关联裁决 | 关联记录 | 对应人工裁决 |
| 责任团队 | 单选/人员 | 来自裁决结果 |
| 处理动作 | 多行文本 | 实际采取的动作 |
| SLA 截止时间 | 日期时间 | 用于暴露等待黑洞 |
| 处理状态 | 单选 | 待处理、处理中、待回访、已关闭、逾期 |
| 完成时间 | 日期时间 | 实际完成时间 |
| 客户回访结果 | 单选 | 满意、一般、不满意、未联系上 |
| 同类投诉 7 天复现 | 数字 | 后续可由统计或人工填入 |
| 是否需要更新规则 | 复选框 | 触发经验沉淀 |
| 复盘结论 | 多行文本 | 人或 AI 总结 |

关键视图：

- 今日待处理。
- 逾期未处理。
- 待回访。
- 需要规则更新。

### 表 5：规则与异常库

用途：保存回路记忆。

| 字段 | 类型 | 说明 |
|---|---|---|
| 规则编号 | 自动编号/文本 | 如 RULE-0001 |
| 规则类型 | 单选 | 分类规则、升级规则、赔付边界、话术建议、异常模式 |
| 触发条件 | 多行文本 | 什么时候使用 |
| 推荐动作 | 多行文本 | 触发后怎么做 |
| 来源记录 | 关联记录 | 来自哪次投诉或复盘 |
| 生效状态 | 单选 | 草稿、试运行、生效、停用 |
| 最近验证结果 | 单选 | 有效、需调整、失效 |
| 版本号 | 文本 | v0.1、v0.2 |

## 7. 工作流设计

### 工作流 1：新增投诉后结构化

触发：

- “投诉原始记录”新增记录，且“当前状态=待结构化”。

动作：

1. 调用 AI Agent：读取“原始投诉内容”和“附件证据说明”。
2. 写回“AI 提取摘要”“客户情绪强度”“是否重复投诉”。
3. 新增一条“AI 归因建议”记录。
4. 将原投诉状态更新为“待归因”或“待裁决”。

### 工作流 2：归因建议后分流

触发：

- “AI 归因建议”新增或更新。

条件：

- 若“置信度 >= 80 且 风险等级=低 且 历史相似案例不为空”，创建“人工裁决”记录，默认裁决动作可为“确认 AI 建议”，等待人一键确认。
- 若“置信度 < 80 或 风险等级=高 或 历史相似案例为空”，创建“人工裁决”记录，并发送飞书消息给裁决人。

### 工作流 3：人工确认后推送修复

触发：

- “人工裁决”中“是否允许自动推送=true”。

动作：

1. 创建“修复与验证”记录。
2. 写入责任团队、处理动作、SLA 截止时间。
3. 将原投诉状态更新为“已推送修复”。
4. 通知责任团队。

### 工作流 4：处理完成后回访

触发：

- “修复与验证”中“处理状态=待回访”。

动作：

1. 给回访负责人发送提醒。
2. 回访结果填入后，若“不满意”或“同类投诉 7 天复现 > 0”，勾选“是否需要更新规则”。
3. 需要更新规则时，创建“规则与异常库”草稿记录。

## 8. AI Agent 提示词草案

### Agent 1：投诉结构化智能体

角色：

你是客户投诉回路中的“感知节点”。你的任务不是替人做最终判断，而是保留原始语境，并把投诉内容转换成下游可处理的结构化信息。

输入：

- 客户原始投诉内容
- 来源渠道
- 涉及产品/服务
- 附件证据说明

输出：

- 100 字以内摘要
- 客户情绪强度：低/中/高
- 关键事实列表
- 客户明确诉求
- 缺失信息
- 是否疑似重复投诉

约束：

- 不得删除对判断有影响的原始事实。
- 不得把不确定内容写成确定结论。
- 如果证据不足，必须写入“缺失信息”。

### Agent 2：归因与动作建议智能体

角色：

你是投诉分类与归因回路中的“归因节点”。你根据结构化投诉信息，给出可审计的分类、根因假设和处理建议。

输入：

- 投诉摘要
- 关键事实
- 客户诉求
- 历史规则与异常库

输出：

- 投诉类型
- 根因假设
- 建议处理动作
- 历史相似案例或“未找到”
- 置信度 0-100
- 风险等级：低/中/高
- 是否需要人工裁决
- AI 不确定点

约束：

- 每条建议必须附证据。
- 高赔付、高舆情、高法律风险必须标为高风险。
- 置信度低于 80 必须进入人工裁决。

### Agent 3：复盘与规则更新智能体

角色：

你是修复与验证回路中的“学习节点”。你根据处理结果、回访结果和复现情况，判断是否需要更新规则。

输入：

- 原始投诉
- 归因建议
- 人工裁决
- 处理动作
- 回访结果
- 同类投诉复现情况

输出：

- 本次处理是否有效
- 原归因是否准确
- 是否需要更新规则
- 建议新增或修改的规则草案
- 下一轮应关注的风险信号

约束：

- 不把一次偶发事件上升为通用规则。
- 如果样本不足，只给“观察项”，不直接给“生效规则”。

## 9. 多智能体自动配置设计

Base 可以承载多智能体，但首版不要做成“完全自动生成多智能体系统”。更稳的做法是：用一张“智能体注册表”把角色、输入、输出、边界和触发条件结构化，再让工作流按注册表调用不同 AI Agent 节点。

### 9.1 自动化分级

| 级别 | 能力 | 首版是否采用 | 原因 |
|---|---|---|---|
| L1：手工配置 | 人手工建表、建字段、写提示词、建工作流 | 可作为兜底 | 最稳定，但不够体现智能体能力 |
| L2：模板化半自动 | 预置表结构、Agent 职责、提示词、工作流路径，人确认后启用 | 推荐 | 可演示“自动设置多智能体”，同时保留治理边界 |
| L3：自然语言全自动生成 | 用户一句话生成表、Agent 和工作流 | 不做首版 | 容易产生字段不一致、权限不清、责任边界模糊 |
| L4：自进化多智能体 | 系统根据运行数据自动新增或改写 Agent | 不做首版 | 风险高，需要审计、版本和回滚机制 |

首版目标定为 L2：系统自动生成多智能体配置草案，人确认后启用。

### 9.2 新增表：智能体注册表

用途：把多个 AI Agent 当成可治理的组织成员，而不是散落在工作流里的黑盒节点。

| 字段 | 类型 | 说明 |
|---|---|---|
| 智能体编号 | 自动编号/文本 | 如 AGENT-001 |
| 智能体名称 | 文本 | 投诉结构化智能体、归因建议智能体、复盘更新智能体 |
| 所属回路 | 单选 | 分类与归因、修复与验证、规则进化 |
| 节点角色 | 单选 | 感知、归因、裁决辅助、执行辅助、验证、学习 |
| 触发条件 | 多行文本 | 什么情况下调用 |
| 输入字段 | 多选/多行文本 | 从哪些表字段读取 |
| 输出字段 | 多选/多行文本 | 写回哪些表字段 |
| 可调用记忆 | 关联记录 | 可读取哪些规则、案例或异常 |
| 禁止动作 | 多行文本 | 不能承诺赔付、不能关闭投诉、不能修改生效规则等 |
| 人工确认门槛 | 多行文本 | 低置信度、高风险、无证据时必须人审 |
| 提示词版本 | 文本 | v0.1、v0.2 |
| 启用状态 | 单选 | 草稿、试运行、生效、停用 |

这张表的价值是：演示时可以直接告诉观众，智能体不是“外挂工具”，而是被放进治理结构里的节点。

### 9.3 自动生成顺序

自动配置不从“生成 Agent”开始，而从回路结构开始。

1. 生成表结构：投诉原始记录、AI 归因建议、人工裁决、修复与验证、规则与异常库、智能体注册表。
2. 生成智能体注册表：写入 3 个默认 Agent，以及各自输入、输出和禁止动作。
3. 生成提示词草案：把第 8 节的提示词写入对应 Agent 的配置说明。
4. 生成工作流草案：新增投诉 -> 结构化 Agent -> 归因 Agent -> 人工裁决 -> 修复验证 -> 复盘 Agent。
5. 人工确认：确认字段、责任人、风险门槛和通知对象。
6. 启用工作流：先试运行，不直接作为正式业务流程。

### 9.4 多智能体协作协议

| 协作关系 | 上游智能体 | 下游智能体/人 | 交接协议 |
|---|---|---|---|
| 感知到归因 | 投诉结构化智能体 | 归因建议智能体 | 必须提供原文、摘要、关键事实、缺失信息 |
| 归因到裁决 | 归因建议智能体 | 人工裁决人 | 必须提供类型、根因、证据、置信度、风险等级 |
| 裁决到修复 | 人工裁决人 | 修复责任团队 | 必须提供最终责任团队、处理动作和 SLA |
| 修复到学习 | 修复责任团队 | 复盘更新智能体 | 必须提供处理结果、回访结果和复现数据 |
| 学习到规则 | 复盘更新智能体 | 回路负责人 | 只能生成规则草案，不能直接生效 |

### 9.5 人工确认门槛

以下动作必须人审：

- 改变客户承诺。
- 涉及赔付、法律、舆情或高价值客户。
- AI 置信度低于 80。
- AI 没有找到历史证据。
- 复盘智能体建议把草案规则改为生效规则。

以下动作可以自动执行：

- 从投诉原文提取结构化字段。
- 生成摘要、关键事实和缺失信息。
- 创建待裁决记录。
- 创建修复与验证记录草案。
- 标记需要规则更新。

### 9.6 演示口径

对外讲法：

> 这不是让 AI 自己管理 AI，而是把多个智能体放进一条业务回路里。Base 负责共享状态，工作流负责编排，AI Agent 负责不同节点的判断，人负责边界和最终承诺。

10 分钟演示时，可以把“智能体注册表”作为关键一页展示：

- 这条回路里有哪些智能体。
- 每个智能体读什么、写什么。
- 哪些动作必须人确认。
- 哪些运行结果会回到规则库。

## 10. 看板设计

### 看板 1：回路健康度

核心指标：

- 今日新增投诉数
- 待结构化数量
- 待裁决数量
- 平均裁决等待时间
- 逾期未处理数量
- 回访完成率
- 同类投诉 7 天复现数
- 规则更新数量

### 看板 2：断点改善

| 断点 | 指标 | 演示解释 |
|---|---|---|
| 信息衰减 | 结构化字段完整率 | 原始投诉是否被保留并转成下游可读字段 |
| 等待黑洞 | 平均裁决等待时间 | 人是否从逐条搬运退到异常裁决 |
| 校验真空 | 回访完成率、复现率 | 结果是否回到下一轮规则 |

## 11. 演示脚本

总时长：8-10 分钟。

### 第 1 分钟：展示旧流程

展示一条传统投诉流程：

客户投诉 -> 客服记录 -> 主管筛选 -> 经理指派 -> 处理 -> 回访

指出三个断点：

- 客服摘要造成信息衰减。
- 主管和经理排队造成等待黑洞。
- 工单关闭后没有验证，形成校验真空。

### 第 2-3 分钟：新增一条投诉

在“投诉原始记录”中新增演示投诉：

客户说已经连续三次反馈售后安装延迟，每次客服都说会催促，但没有人解释原因，也没有明确上门时间。客户要求今天给出解决方案，否则会在社交平台公开投诉。

### 第 4-5 分钟：AI 自动结构化与归因

展示 AI 写回的字段：

- 情绪强度：高
- 投诉类型：交付延迟
- 根因假设：安装资源排期与客服承诺不同步
- 风险等级：高
- 需要人工裁决：是
- 不确定点：缺少订单区域、安装服务商、历史预约记录

讲解重点：

AI 没有替人做最终承诺，它把人需要裁决的信息准备好了。

### 第 6-7 分钟：人工裁决

裁决人选择：

- 裁决动作：升级处理
- 最终责任团队：供应链/服务交付
- 是否允许自动推送：是

讲解重点：

人从“流程搬运者”变成“边界裁决者”。

### 第 8-9 分钟：修复与验证

工作流创建修复记录，责任团队收到通知。

演示一条已完成样例：

- 处理状态：待回访
- 客户回访结果：一般
- 同类投诉 7 天复现：3
- 是否需要更新规则：是

讲解重点：

流程到这里不是结束，反馈进入下一轮规则。

### 第 10 分钟：看板收束

展示回路健康度看板：

- 哪些投诉还在等人。
- 哪些 AI 建议被人工改判。
- 哪些规则需要更新。
- 这条回路是否比上一轮更聪明。

## 12. 验收清单

首版样板完成后，应能通过以下检查：

- 新增一条投诉后，AI 能生成结构化摘要和归因建议。
- 高风险或低置信度记录能进入人工裁决视图。
- 人工确认后能生成修复与验证记录。
- 处理完成后能记录回访结果。
- 复现或不满意结果能触发规则更新草稿。
- 看板能展示至少三个回路健康指标。
- 智能体注册表能说明每个 Agent 的输入、输出、禁止动作和人工确认门槛。
- 多智能体之间的每次交接都有结构化字段承接，不靠口头解释。
- 现场演示不依赖真实客户数据。

## 13. 不做的事

首版不做：

- 与真实客服系统、CRM、电话转写系统集成。
- 自动赔付、自动承诺客户结果。
- 全行业通用模板库。
- 复杂权限矩阵。
- Loop Designer 双向同步。
- 自主新增或改写生效中的 AI Agent。

这些都可以作为第二阶段，但不应该进入首版示范。

## 14. 后续路线

### 第 1 阶段：Base 样板

手工创建 Base、表、字段、视图和示例数据。用工作流和 AI Agent 跑通 10 分钟演示，同时用“智能体注册表”展示多智能体配置。

### 第 2 阶段：课程现场模板

把样板复制成课程模板，现场让学员替换为自己的业务场景：销售线索、合同审核、教研讲评、交付异常。

### 第 3 阶段：接入 Loop Designer

Loop Designer 输出回路方案后，一键生成 Base 表结构、字段、智能体注册表、Agent 提示词和工作流草案。Base 负责持续运行记录，Loop Designer 负责沉淀回路资产。
