# Loop Designer 下一版本 PRD：三重对齐与成熟度诊断层 v0.2

## 0. 本轮修订说明

本版整合三方审阅意见，重点修正四个问题：

- 诊断到行动之间不能断开。用户看到短板后，必须知道先改什么、怎么改、改完如何确认变好。
- 方法论术语必须有业务翻译层。用户不应先理解“三重对齐”“生态进化性”等概念，才能使用产品。
- 评分必须可信。用户端只展示层级、证据和置信度，不把 0-100 原始分数作为主展示。
- 产品需要 Aha Moment。生成方案后自动给出一句话诊断、最大短板和推荐优先行动，让用户立刻感到“这不是一份 AI 咨询稿，而是一条可衡量的进化路径”。

本版的核心产品判断：

> 传统组织诊断看整体，Loop Designer 看每一条真实业务回路；传统诊断靠问卷，Loop Designer 从设计方案反推成熟度和下一步行动。

## 1. 背景

业务回路设计正在成为碳硅组织进化的核心工作。当前 Loop Designer 已经能把组织输入转成五阶段回路方案，并输出人类角色、智能体角色、系统角色、接口契约、RACI、启动检查和 30 天路线。

下一版本要补齐方法论闭环：让每条回路不仅“能被设计出来”，还要能被判断是否对齐、是否成熟、下一步该补哪里。

参考输入来自《AI时代组织成熟度模型 — 产品迭代设计参考》。其中最重要的产品判断是：成熟度模型不应成为新的测评工具，而应成为回路设计器内部的诊断层，从设计数据中推导结果，做到可算、可视、可行动。

## 2. 产品定位

下一版本定位为：

> Loop Designer = 业务回路设计台 + 三重对齐检查器 + 成熟度诊断层。

它不新增独立测评入口，不要求用户额外填写问卷。诊断结果从现有回路设计数据中推导，并附带证据链、短板和行动建议。

对外表达可以收敛为一句话：

> 让你的组织回路，从设计图变成可衡量的进化路径。

产品差异化不在“又多一个成熟度评分”，而在“生成即诊断”。用户完成一条回路设计后，系统自动回答三个问题：

- 这条回路能不能跑？
- 哪里最容易出问题？
- 下一步先修什么？

## 3. 目标用户

第一优先用户是企业内部组织设计者、业务负责人和咨询顾问。他们需要在一个具体业务回路上判断：

- 这条回路是否服务战略目标。
- 人与 AI 的边界是否清楚。
- 接口、数据、验证机制是否闭环。
- 这条回路当前处于什么成熟度。
- 未来 30 天最该补哪几个缺口。

## 4. 成功标准

本版本完成后，用户在生成一条回路方案后，应能直接看到：

- 一句业务语言诊断，例如“这条回路已经能试点，但反馈闭环不足，暂时不适合直接规模化”。
- 一组三重对齐结果：目标对齐、价值对齐、逻辑对齐。
- 一个五维成熟度画像：回路成熟度、三重对齐度、编排能力、智能密度、生态进化性。
- 每个评分背后的证据链。
- 当前最大短板。
- 1 个推荐优先行动，以及 3-5 条可执行升级建议。
- 每条 critical 建议都能说明“不改的风险”，并绑定下一步操作。
- 用户能从 critical 建议一键进入定向优化。
- 定向优化后自动重新评估，并显示变化摘要。
- 导出的 Markdown、PDF、飞书文档中保留这些诊断结果。
- 提交到 Matrix Origin 的设计提案中带上 `methodologyAnalysis`，但不自动修改 Matrix 拓扑、权限或接口。

## 5. 非目标

本版本不做：

- 独立成熟度测评工具。
- 新增一套问卷流程。
- 组织级多回路聚合仪表盘。
- Matrix Origin SIGNALS / BECOMING 的完整改造。
- 数据库级成熟度快照表。
- 可配置权重后台。

这些能力可作为后续版本进入 Matrix Origin 或组织级诊断模块。

## 6. 方法论模型

### 6.1 三重对齐

三重对齐沿用《碳硅组织》术语边界：

| 维度 | 判断问题 | 当前可映射字段 |
|---|---|---|
| 目标对齐 | 这条回路是否服务明确战略目标，并能被业务结果验收 | `executiveSummary`, `valueFlow`, `stages.successSignal`, `governance.kpis`, `roadmap.checkpoint` |
| 价值对齐 | 人和 AI 的边界是否清楚，谁判断、谁承诺、谁接管异常 | `humanRoles`, `agentRoles`, `hitlNodes`, `stageAccountabilities`, `launchReadiness` |
| 逻辑对齐 | 数据、接口、验收、审计和验证问题是否形成闭环 | `systemRoles`, `interfaces`, `governance.lifecycleRule`, `validationQuestions` |

用户端展示时不直接抛出术语，应使用“业务解释 + 方法论标签”的组合：

- 目标对齐：这条回路是不是在服务真正重要的业务目标。
- 价值对齐：人和 AI 的边界是不是清楚。
- 逻辑对齐：数据、接口和验收是不是能闭环。

### 6.2 五维成熟度

| 维度 | 含义 | 推导线索 |
|---|---|---|
| 回路成熟度 | 五阶段是否完整，反馈是否能改进下一轮 | 五阶段完整性、反馈阶段质量、生命周期规则、复盘节奏 |
| 三重对齐度 | 目标、价值、逻辑三类对齐是否同时成立 | 三重对齐评分聚合 |
| 编排能力 | 人、智能体、系统和接口是否被清楚编排 | 角色覆盖、RACI、接口契约、HITL、启动检查 |
| 智能密度 | AI 是否进入关键节点并沉淀可复用能力 | AI 参与度、智能体数量、工具、上下文来源、可审计输出 |
| 生态进化性 | 这条回路是否能被迭代、扩散和纳入组织演化 | 版本记录、30 天路线、验证问题、Matrix 提案、生命周期规则 |

五个维度的用户端翻译：

- 回路成熟度：这条业务链路是不是形成了完整闭环。
- 三重对齐度：目标、边界和验证是不是互相支持。
- 编排能力：人、AI、系统和接口是不是各就各位。
- 智能密度：AI 是不是进入了关键节点，而不只是外围辅助。
- 生态进化性：这条回路是不是能复盘、迭代、扩散到下一条回路。

成熟度层级采用五级：

- L1 工具外挂
- L2 试点接入
- L3 规模化应用
- L4 系统整合
- L5 生态进化

用户端展示时必须带业务语言解释：

| 层级 | 标签 | 业务语言解读 | 典型画像 | 升级代价 |
|---|---|---|---|---|
| L1 | 工具外挂 | AI 只是辅助工具，还没进入业务流程 | 用 AI 写材料、查资料，但流程仍靠人手工推动 | 低：选定 1 个高频场景做小试点 |
| L2 | 试点接入 | 已验证局部价值，但还不能稳定复用 | 某条客服、销售或交付回路用了 AI，但角色和数据还没打通 | 中：补齐责任边界和反馈机制 |
| L3 | 规模化应用 | AI 进入多条关键回路，开始形成组织级复用 | 3 条以上回路有 AI 角色、验收信号和复盘机制 | 中高：标准化接口、权限和验证方式 |
| L4 | 系统整合 | 多回路之间的数据、接口和角色能互相协同 | 回路间共享规则、Prompt、数据对象和异常处理机制 | 高：需要组织级架构与治理设计 |
| L5 | 生态进化 | 组织能持续发现张力，并让回路自我进化 | 有张力检测、变更提案、版本管理和持续优化机制 | 持续：需要工具、文化和治理共同沉淀 |

## 7. 核心用户流程

### 7.1 新建或继续回路

用户仍按现有流程完成诊断、蓝图和回路设计。流程不增加额外问卷。

### 7.2 生成回路方案

模型输出原有 `LoopPlan`，并新增：

- `maturityMapping`

生成失败或模型漏字段时，由结构校验和修复流程兜底。

### 7.3 查看诊断

方案页新增“对齐与成熟度”面板，位置放在“回路总览”之后。面板默认展开，但用户可以折叠，避免打断只想先看方案的用户。

首次生成方案后的 Aha Moment：

> 系统显示“你的回路已自动完成成熟度诊断”，并给出一句话诊断、最大短板和推荐优先行动。

面板包含：

- 综合成熟度标签。
- 一句话诊断。
- 亮点维度。
- 五维成熟度条形图或雷达式简化视图。
- 三重对齐三列卡片。
- 关键短板。
- 推荐优先行动。
- 升级建议，critical 建议必须展示“不改的风险”。
- 证据链展开区。
- “AI 评估，仅供参考”标签和反馈入口。

### 7.4 定向优化

右侧定向优化选项新增：

- 修复业务目标不清
- 修复人机边界问题
- 修复数据闭环问题
- 提升回路成熟度
- 应用诊断建议到 30 天路线

用户可以要求“把这条回路从 L2 推到 L3”或“修复数据闭环问题”，模型应保留方案主体，只重写相关字段。

定向优化闭环：

1. 用户点击推荐优先行动。
2. 系统自动填入定向优化指令。
3. 模型只改相关字段，不重写无关模块。
4. 新版本生成后自动重新评估。
5. 页面显示变化摘要，例如“回路成熟度 L2 → L3；智能密度不变；最大短板从数据闭环转为角色边界”。

所有优化版本保留在现有版本记录中，方便用户回退或对比。

### 7.5 导出和提交 Matrix

Markdown、PDF、飞书导出增加“对齐与成熟度诊断”章节。

提交 Matrix Origin 时，`methodologyAnalysis` 应包含：

- 三重对齐结果。
- 五维成熟度结果。
- 证据链。
- 短板与建议。

Matrix Origin 只把这些内容作为审阅证据，不自动扩大权限、不自动新增 Cell、不自动发布版本。

### 7.6 30 天后重新评估

本版本不做完整健康度定时检查，但必须预留回访机制：

- 30 天路线的最后一个 checkpoint 增加“重新评估成熟度”提示。
- 导出内容保留“30 天后建议复评”。
- 后续可接入提醒、演进时间线和版本快照。

## 8. 数据结构

### 8.1 共享类型

建议在 `packages/types/src/index.ts` 或后续拆分文件中增加以下类型，再由 Loop Designer 使用。

```ts
export type AlignmentDimension = "goal" | "value" | "logic";

export type MaturityDimension =
  | "loop_maturity"
  | "triple_alignment"
  | "orchestration"
  | "intelligence_density"
  | "eco_evolution";

export type MaturityLevel = 1 | 2 | 3 | 4 | 5;

export type EvidenceItem = {
  source: string;
  summary: string;
  userLabel: string;
  strength: "strong" | "partial" | "missing";
  confidence: "high" | "medium" | "low";
  gap?: string;
};

export type AlignmentScore = {
  dimension: AlignmentDimension;
  score: number;
  level: MaturityLevel;
  userExplanation: string;
  evidence: EvidenceItem[];
  gap?: string;
  recommendation?: string;
};

export type MaturityScore = {
  dimension: MaturityDimension;
  level: MaturityLevel;
  score: number;
  userExplanation: string;
  evidence: EvidenceItem[];
  bottleneck?: string;
};

export type UpgradeSuggestion = {
  dimension: MaturityDimension | AlignmentDimension;
  priority: "critical" | "important" | "optional";
  action: string;
  actionType: "regenerate_field" | "add_field" | "apply_to_roadmap" | "manual";
  targetField?: string;
  riskIfIgnored?: string;
  expectedEffect: string;
};

export type LoopMaturityMapping = {
  assessedAt: string;
  assessmentMode: "model_primary" | "algorithm_primary" | "hybrid";
  alignment: AlignmentScore[];
  maturity: MaturityScore[];
  overallLevel: MaturityLevel;
  oneLineDiagnosis: string;
  highlightDimensions: MaturityDimension[];
  bottlenecks: string[];
  recommendedAction: UpgradeSuggestion;
  upgradeSuggestions: UpgradeSuggestion[];
  calibrationFeedback?: {
    userRating?: "too_high" | "accurate" | "too_low";
    note?: string;
  };
};
```

字段说明：

- `score` 保留给内部校准、测试和后续算法优化，用户端默认不展示原始分数。
- `userLabel` 用于展示“有支撑 / 部分支撑 / 证据不足”等用户友好标签。
- `gap` 在 `partial` 或 `missing` 时必填，避免用户只看到模糊的“部分支撑”。
- `actionType` 让建议能绑定产品动作，critical 建议必须能触发一键定向优化或明确说明需要人工处理。
- `riskIfIgnored` 只对 critical 建议必填，用业务语言说明不改会发生什么。

### 8.2 LoopPlan 扩展

`LoopPlan` 增加可选字段：

```ts
maturityMapping?: LoopMaturityMapping;
```

先使用可选字段，保证旧方案、历史会话和旧测试数据向后兼容。

## 9. 评分规则

本版本采用混合策略：

- 生成时要求模型输出结构化诊断。
- 本地纯函数做基础校验和补算。
- 如果模型诊断缺失，本地函数基于现有字段给出保守评分。

初始规则不做复杂权重配置，内部统一 0-100 分，并映射到 L1-L5：

- 0-39：L1
- 40-54：L2
- 55-69：L3
- 70-84：L4
- 85-100：L5

用户端默认不展示原始分数，只展示：

- 层级。
- 业务语言解释。
- 支撑证据。
- 置信度。
- 缺口。
- 推荐动作。

综合成熟度取五维中的最低层级，避免平均分掩盖短板。但 UI 必须先展示亮点，再解释“本回路受 XX 维度限制”，降低用户挫败感。

可选提供内部调试字段或后续管理端视图展示加权参考分，但不作为主要产品展示。

### 9.1 证据标签

| 内部值 | 用户端标签 | 含义 | 必填补充 |
|---|---|---|---|
| `strong` | 有支撑 | 当前方案中有明确字段支撑 | 来源字段和摘要 |
| `partial` | 部分支撑 | 方案有相关线索，但不足以确认闭环 | 具体缺口 |
| `missing` | 证据不足 | 当前方案缺少判断所需信息 | 具体要补什么 |

证据链展开时，应展示来源字段，例如：

> 来源：`interfaces.acceptanceCriteria`。推导：接口有验收标准，因此逻辑对齐具备部分支撑。缺口：缺少失败后的复盘机制。

## 10. UI 要求

### 10.1 视觉原则

沿用当前产品的深色工作台、cyan、acid green、signal orange，不新增大面积装饰色。

### 10.2 组件

新增 `MaturityAssessmentPanel`：

- 顶部显示综合层级，例如 `L2 试点接入`。
- 顶部展示一句话诊断，例如“这条回路已经适合试点，但反馈闭环不足，暂时不适合直接扩圈”。
- 展示“帮你回答：能跑吗？哪里最容易出问题？下一步该修什么？”的价值锚点。
- 显示五维成熟度，每维包含层级、业务解释、瓶颈，不默认展示原始分数。
- 显示三重对齐，每维包含状态、证据、建议。
- 先展示亮点维度，再展示限制综合层级的短板维度。
- 显示推荐优先行动，critical 动作提供一键定向优化入口。
- 使用可展开证据链，避免首屏信息过载。
- 升级建议按 `critical`、`important`、`optional` 排序。
- 面板默认展开，但允许折叠。
- 证据链展开时标注来源字段、推导摘要和缺口。
- 显示“AI 评估，仅供参考”标签，并提供“评分偏高 / 准确 / 偏低”反馈入口。

### 10.3 推荐优先行动

推荐优先行动是本版本最关键的产品闭环。展示规则：

- 从 `upgradeSuggestions` 中选择第一个 critical 建议。
- 如果没有 critical，则选择影响综合层级的 important 建议。
- 如果建议可自动处理，按钮文案使用业务语言，例如“修复数据闭环问题”。
- 如果建议需要人工处理，按钮文案为“查看人工处理清单”。
- 操作后进入定向优化或展示人工操作步骤。

### 10.4 空状态

如果旧方案没有 `maturityMapping`：

- 显示“发现未诊断的设计方案”。
- 提供“生成对齐与成熟度诊断”的引导，不自动改变原方案。
- 主按钮文案为“立即生成成熟度报告”。

## 11. 生成提示词要求

系统提示词新增规则：

- 必须输出三重对齐诊断。
- 必须输出五维成熟度诊断。
- 每个评分必须带证据链。
- 必须输出一句话诊断。
- 必须输出亮点维度、关键短板和推荐优先行动。
- 评分只能基于用户输入和方案字段，不得虚构精确数据。
- 如果信息不足，应标记为 `partial` 或 `missing`，并写清具体缺口。
- critical 建议必须带 `riskIfIgnored`。
- `action` 必须使用业务语言，不得只写“补齐逻辑对齐”这类方法论术语。
- `actionType` 必须绑定可执行产品动作。
- 不得把成熟度诊断写成泛泛咨询话术。

`OUTPUT_SHAPE` 同步增加 `maturityMapping`。

## 12. 导出要求

`planToMarkdown` 新增章节：

```md
## 对齐与成熟度诊断

### 诊断摘要

### 三重对齐

### 五维成熟度

### 推荐优先行动

### 关键短板

### 升级建议
```

导出要求：

- 文档开头增加 2-3 句话的诊断执行摘要，适合高管快速阅读。
- 成熟度诊断章节不展示原始分数，只展示层级、证据和建议。
- critical 建议必须带“不改的风险”。
- 保留“30 天后建议重新评估”的提醒。
- PDF 和飞书导出沿用 Markdown 内容，不单独设计一套结构。
- 后续可增加“一页纸高管简报”导出模式，本版本先在 Markdown 中预留结构。

## 13. Matrix Origin 集成

现有 `CircuitDesignStudyPayload.methodologyAnalysis` 保持为承载点。

下一版本提交内容应包含：

```ts
methodologyAnalysis: {
```

- 如果建议涉及跨 Cell Interface、新 Actor 或权限扩大，只能进入人工确认队列。

## 14. 验收标准

### 14.1 单元测试

- `parsePlan` 能解析包含 `maturityMapping` 的方案。
- 旧版方案不含 `maturityMapping` 时仍能解析。
- 本地成熟度评估函数能根据样例方案输出五维评分。
- 综合层级取最低维度。
- 用户端格式不展示原始 0-100 分数。
- `partial` 和 `missing` 证据必须包含 `gap`。
- critical 建议必须包含 `riskIfIgnored` 和 `actionType`。
- Markdown 导出包含“对齐与成熟度诊断”。

### 14.2 页面验收

- 生成后方案页出现成熟度面板。
- 首屏可见一句话诊断、最大短板和推荐优先行动。
- 面板在桌面和移动端不遮挡、不溢出。
- 证据链可展开查看。
- 证据链展示来源字段、推导摘要和缺口。
- critical 建议可一键进入定向优化或展示人工处理清单。
- 定向优化后自动重新评估，并展示变化摘要。
- 旧方案显示清晰空状态。
- 定向优化入口使用业务语言，不直接暴露方法论黑话。

### 14.3 集成验收

- 提交 Matrix Origin 的 payload 中包含成熟度诊断。
- Matrix 映射仍保持人工确认边界。
- 导出的 Markdown、PDF、飞书文档内容一致。
- 导出文档包含 2-3 句诊断执行摘要。
- 30 天路线中保留重新评估提醒。

## 15. 推荐实施顺序

1. 定义共享类型和 `LoopPlan` 可选字段。
2. 增加本地成熟度评估纯函数和测试。
3. 更新生成提示词、输出形状和结构解析测试。
4. 增加 `MaturityAssessmentPanel`，先完成一句话诊断、最大短板和推荐优先行动。
5. 增加定向优化动作绑定和自动重评估。
6. 增加证据链展开、置信度和反馈入口。
7. 更新 Markdown / PDF / 飞书导出。
8. 更新 Matrix 提交的 `methodologyAnalysis`。
9. 补充端到端手工验收。

优先级原则：

- P0 必须先完成“诊断到行动闭环”，再做更复杂的图表。
- 原始分数不进入首版 UI。
- 先实现一键定向优化，再考虑分享卡片、同行对标和历史趋势。

## 16. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 评分看起来武断 | 每个评分必须有证据链和缺口说明 |
| 模型输出泛化 | Schema、prompt 和本地补算共同约束 |
| 旧会话不兼容 | 新字段保持可选 |
| Matrix 集成范围失控 | 只透传诊断，不改拓扑 |
| UI 信息过载 | 默认展示摘要，证据链折叠 |
| 用户看完不行动 | 首屏提供推荐优先行动和一键定向优化 |
| 方法论术语劝退业务用户 | 所有用户端文案使用业务语言，方法论标签放在次级位置 |
| 木桶效应造成挫败感 | 先展示亮点，再解释受限维度和升级代价 |
| 用户不信任 AI 判断 | 标注“AI 评估，仅供参考”，提供反馈入口 |
